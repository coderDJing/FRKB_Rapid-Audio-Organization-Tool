# Mixtape Stem Fast 激进提速方案草案（ONNX/DirectML 路线）

更新时间：2026-03-03（草案 v0.1）

## 背景与问题
1. 当前实时档位 `fast` 与导出档位 `quality` 都基于 `htdemucs`，仅参数不同，速度差距有限。
2. 在仅 CPU 或“核显可见但推理后端不可用”的机器上，分离会回落到 CPU，单曲耗时明显偏长。
3. 目标不是替代 `quality`，而是把 `fast` 定位为“实时可用、快速出结果”的预览档位。

## 文档定位
1. 本文仅定义 `fast` 激进提速技术路线。
2. 本文不决定“导出时是否强制 quality”。
3. 本文优先保证可落地与可回退，不做一次性不可控重构。

## 目标与非目标
1. 目标：`fast` 平均耗时较当前实现降低 `50%~75%`（不同硬件允许波动）。
2. 目标：在 Windows Intel/AMD 核显环境优先尝试 DirectML 加速。
3. 目标：`quality` 链路保持可用、行为稳定、结果口径不变。
4. 非目标：本阶段不追求 `fast` 音质与 `quality` 完全一致。
5. 非目标：本阶段不引入在线模型下载与模型市场。

## 方案总览（双引擎分层）
1. `quality`：保留现有 Demucs CLI（`htdemucs`）链路。
2. `fast`：新增轻量推理引擎链路（优先 ONNX Runtime + DirectML）。
3. 运行时按 profile 分流，不再让 `fast/quality` 仅靠参数差异。
4. 新链路失败时可回退到当前 Demucs `fast`，确保功能可用。

## Fast 引擎候选与建议
1. 候选 A：MDX 类 4-stem ONNX 模型（推荐优先验证）。
2. 候选 B：更小参数量的 Demucs 变体导出 ONNX（次选）。
3. 建议：先落地 1 个稳定模型，避免多模型并发导致调参爆炸。
4. 模型准入标准：
5. 支持 44.1k 采样率输入。
6. 输出 4 stem（vocal/inst/bass/drums）或可稳定映射到现有字段。
7. 在目标机器上可稳定连续处理 20 首以上不中断。

## 推理后端策略
1. Windows 优先：`DirectML -> CPU`。
2. macOS/Linux 首期：`CPU`（后续可评估 CoreML/Metal、CUDA 路线）。
3. 设备探测与回退需要落日志，便于定位“为什么没走加速”。

## 处理链路（Fast）
1. 输入解码：统一到 `Float32 PCM`（建议 44.1k，立体声）。
2. 分块推理：固定窗口 + overlap，减少显存/内存峰值。
3. 拼接策略：窗函数融合，控制块边界伪影。
4. 输出写盘：落现有 stem 缓存目录结构，复用后续播放与波形链路。
5. 缓存命中：沿用现有 key 机制，并新增 `engine/model/backend` 维度。

## 质量预期（Fast 是否“够用”）
1. 预期可用：实时混音、段落切换、快速编排与试听决策。
2. 预期不足：纯净 acapella 导出、长时间 solo 精听场景。
3. 产品文案建议：明确 `fast` 为“实时预览档”，避免用户误解为最终成品档。

## 数据与版本口径
1. `stemModel` 建议扩展为可区分引擎与模型：
2. 示例：`onnx-mdx23c@fast`、`htdemucs@quality`。
3. `stemVersion` 建议编码引擎版本：
4. 示例：`stem-engine-onnx-20260303`。
5. 保持与当前缓存隔离，防止旧缓存误复用。

## 兼容与回退策略
1. Fast 新链路失败：自动回退 Demucs fast。
2. 连续失败阈值：同曲目/同会话内失败超过阈值后暂停自动重试，避免无效打满 CPU。
3. UI 呈现失败原因：设备不可用、模型缺失、后端初始化失败、推理崩溃。

## 实施阶段（建议）
1. 阶段 F1：引擎骨架接入
2. 新增 `StemFastEngine` 抽象层，打通 `fast` profile 分流。
3. 保留现有 Demucs 全链路不动，确保可回退。
4. 阶段 F2：ONNX Runtime + DirectML 首版
5. 接入单模型推理、分块拼接、写盘与缓存落库。
6. 打通运行时探测、日志与失败回退。
7. 阶段 F3：性能与稳定性调参
8. 调整 chunk/overlap/batch/thread，完成基准测试矩阵。
9. 阶段 F4：体验收敛
10. 增加 profile 文案、状态提示、错误文案、埋点字段。

## 验收标准（草案）
1. 功能正确性：
2. `fast`、`quality` 均可产出可播放 4 stems 文件。
3. 混音时间线加载、播放、导出前检查逻辑不回归。
4. 性能：
5. 在目标 CPU+核显机器，`fast` 中位耗时较当前基线下降至少 `50%`。
6. 在纯 CPU 回退路径，耗时不劣于当前 `fast` 超过 `10%`。
7. 稳定性：
8. 连续 20 首批处理无崩溃、无僵死进程、无明显内存泄漏。

## 观测与日志建议
1. 每首记录：引擎、模型、后端、设备、总耗时、推理耗时、I/O 耗时、回退次数。
2. 关键失败码标准化：`FAST_BACKEND_UNAVAILABLE`、`FAST_MODEL_MISSING`、`FAST_INFER_FAILED`。
3. 日志需可定位“为什么没走 DirectML”与“为什么回退到 CPU”。

## 风险清单
1. ONNX 模型授权与分发体积风险。
2. DirectML 在不同驱动版本上的兼容性波动。
3. 分块拼接参数不当导致边界伪影或泵音。
4. 多引擎并存带来维护复杂度上升。

## 待决策项（暂不在本草案定案）
1. 导出时是否强制 `quality`。
2. 是否在空闲时自动补跑 `quality` 缓存。
3. 是否开放“fast 使用 Demucs/ONNX”手动切换开关。

## 建议下一步
1. 先做 F1/F2 原型分支验证一台 Intel 核显与一台 AMD 核显机器。
2. 产出 10 首样本的 A/B 耗时与主观听感对照表。
3. 评审后再决定是否推进“导出强制 quality”策略。
