# 分拣脚本

## 你手工只做这三步

1. 艾特本文。Codex 自动从 `Upan` 取约 500 首到 `test`，并把算法与 Rekordbox 不一致的歌移到
   `needReview`。
2. 你只查看 `needReview`：确认 Rekordbox 对就不动；Rekordbox 不对就手工修正它的 BPM / grid。然后把
   本轮 `test` 中剩下的歌和 `needReview` 中剩下的歌，**全部**放入 `review`。
3. 艾特 [`准备好rkb新样本.md`](./准备好rkb新样本.md)。Codex 从 `review` 取这一整批正确 truth，纳入后续
   训练 / 调参数据。

不用把 `needReview` 移回 `test`，也不要因为算法一致或不一致删掉任何歌；两边的歌最后都要进 `review`。

当你艾特本文时，默认执行一轮新的 Rekordbox 人工 review 前分拣：

`Upan → test → needReview →（你人工 review）→ review`

你不需要复制命令。`test` 和 `needReview` 是本轮的临时工作队列；`review` 是你确认后的整批样本入口。

## Codex 的固定动作

1. 只读 preflight：确认 `test` 为空，`needReview` 不存在或为空，且没有 active sealed batch。这样绝不会把
   两轮样本混在一起。
2. 从 `Upan` 前移 500 首到 `test`，并从 `Upan` 移除同一批条目。
3. 对完整 `test` 逐首运行当前生产算法，与 Rekordbox BPM / grid 比较，生成不可篡改的 dry-run report。
   不允许 `--only` / `--limit` 缩小本轮分母。
4. 若 `errorTrackCount = 0`，只按这份报告写回：不一致的曲目从 `test` 移到 `needReview`；一致曲目继续留在
   `test`。不会重新分析音频。
5. 报告永久记录完整 roster、每首 audio SHA-256、当前 solver/hash 与 label-QA 标记；它只证明
   分拣和写回没有混批或替换音频，不给未来候选贴“已训练/已调参”标签。

固定编排脚本是：

```text
scripts/prep_move_tracks_between_playlists.py
```

它内部调用：

```text
scripts/move_rekordbox_playlist_grid_diffs.py --pre-review
```

默认命令形状如下；日常直接艾特本文即可：

```powershell
& "vendor/demucs/win32-x64/runtime-cpu/python.exe" `
  "scripts/prep_move_tracks_between_playlists.py" `
  --source "Upan" --target "test" --limit 500 --apply --then-triage --triage-apply
```

## 你做什么

详见本文最前面的“你手工只做这三步”。

不要把 `needReview` 的歌移回 `test`，也不要在本轮未完成前往 `test` / `needReview` 混入下一批歌。

## 这轮数据的用途

分拣只用旧算法决定人工检查优先级，不影响未来候选。把 `review` 交给“准备好新样本”后，这批才会作为
development 数据参与训练/调参；从那一刻起，它不能再证明使用过它的同一候选提升。

## 旧数据维护

只有维护已经 `consumed` 的旧 batch 时，才直接使用
`scripts/move_rekordbox_playlist_grid_diffs.py --consumed-maintenance --consumed-batch-id <batchId>`。
它不负责从 `Upan` 取样，也不进入新的 `review` intake。
