# Downbeat Harmonic / Phrase 增强性能评估草案

更新时间：2026-04-07

## 1. 背景

当前项目已经落地了一版较轻量的 `downbeat / bar-level` 后处理，主要依赖：

- 全频能量包络
- 低频能量包络
- onset / low onset
- 每 4 拍候选相位打分

这类方案对鼓点明显的电子舞曲通常已经够用，但在以下场景里仍可能不稳：

- intro 很空，kick 不够明显
- `1` 和 `3` 都很重，鼓组模式太像
- breakbeat / 切分鼓把相位带歪
- 真正的小节边界更多体现在和声变化或乐段变化，而不是鼓

因此后续确实有继续加入 `harmonic / phrase` 特征的价值，但在继续动手前，必须先评估性能成本，避免把分析链路搞成慢得像老牛拉车。

## 2. 当前分析链路的时间预算

`src/main/services/keyAnalysis/types.ts`

当前关键超时预算：

- 分析阶段超时：`60s`
- 整个 job 超时：`3min`

从超时角度看，额外加一点 harmonic 计算通常不会直接把单轨分析打爆，但批量分析时会线性放大总耗时，所以重点风险不是“单次超时”，而是“几十首歌一起分析时总耗时明显变长”。

## 3. 本地微基准结论

本次使用本地 Node 环境对 `90s` 人造音频做了微基准，测的是同一机器下不同方案的大致量级。结论主要用于相对比较，不代表所有用户机器上的绝对耗时。

### 3.1 当前已落地的轻量 downbeat 后处理

测量对象：

- `estimateFirstBeatMsFromPcm`
- `estimateBarBeatOffsetFromPcm`

结果：

- `90s` 音频整体耗时约 `57.48ms`

结论：

- 当前这版非常轻
- 不构成严重性能问题
- 默认全量开启是安全的

### 3.2 假设增加 JS 版 harmonic（Meyda chroma）全长分析

测试方式：

- `Meyda.extract('chroma')`
- `2048 window / 1024 hop`
- 全长 `90s`

结果：

- 约 `653ms` 到 `903ms` 量级

结论：

- 明显比当前轻量方案重一个数量级以上
- 单轨仍可接受
- 但不适合默认对所有轨道、所有场景都跑

### 3.3 压缩后的较便宜配置

测试结果：

- `45s + hop 1024`：约 `323ms`
- `45s + hop 2048`：约 `157ms`
- `30s + hop 2048`：约 `120ms`

结论：

- 如果只在前 `30s` 到 `45s` 做 harmonic 补算，并适当拉大 hop，成本是可以接受的
- 但仍然明显高于当前轻量 downbeat 后处理

## 4. 性能风险判断

### 4.1 当前已落地版本

结论：

- 不会造成严重性能问题

原因：

- 只做包络、低频包络、简单 onset 和 4 相位打分
- 没有 FFT/chroma 这种更重的频域特征
- `90s` 级别样本实测只有几十毫秒

### 4.2 如果直接上 JS 版 harmonic / phrase 全量默认开启

结论：

- 不建议

原因：

- 单轨额外增加约 `0.6s` 到 `1.0s+`
- 批量分析时会线性累积
- mixtape 批量 BPM 分析和后台闲时分析都会明显变慢

### 4.3 如果做成“低置信度时才补算”

结论：

- 可以接受

推荐条件：

- 当前 downbeat 相位评分不够稳时才触发
- 只分析前 `30s` 到 `45s`
- 使用较便宜的 hop 配置，例如 `2048`
- 只输出用于修正 `barBeatOffset` 的补充分数，不替代现有轻量主流程

## 5. 现阶段建议

当前阶段先保持现状，不立即把 harmonic / phrase 真正接入主链路。

推荐策略：

1. 默认继续使用当前轻量 downbeat 后处理
2. 后续如果要继续提精度，优先做“低置信度补算”模式
3. 若补算效果被证明有价值，再考虑：
   - 继续优化 JS 版
   - 或把 harmonic 特征下沉到 Rust / QM 侧，减少 Node 侧额外耗时

## 6. 最终结论

一句话总结：

- 当前已经落地的 downbeat/bar-level 后处理，性能基本没压力
- 继续做 harmonic / phrase 是有价值的，但不能默认全量硬上
- 最合理的路线是：`轻量主流程 + 低置信度时短窗口补算`

也就是说，这事不是不能做，而是不能用最蠢的做法做。
