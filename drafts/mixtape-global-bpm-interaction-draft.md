# Mixtape 自动录制全局 BPM 包络交互草案

## 讨论目标

本草案只讨论 Mixtape 自动录制中 `全局 BPM / BPM 包络` 的交互设计，不讨论实现方案、数据结构和开发细节。

当前需要解决的核心问题是：

- 多首歌曲导入后，原始 BPM 不一致，导致“全局 BPM”在认知上不成立
- 用户想编辑的是一个有明确含义的项目节奏线，而不是一条自由绘制的折线
- BPM 变化点希望与网格线对齐，交互应该服务于编排，而不是服务于鼠标像素位置

## 核心设计结论

### 1. 项目进入编辑态时，先有一个明确的项目初始 BPM

- 项目初始 BPM 默认取第一首歌的 BPM
- 如果第一首歌没有有效 BPM，则取第一首有有效 BPM 的歌曲 BPM
- 如果整个项目都没有有效 BPM，则使用兜底默认值

项目初始 BPM 一旦确定，就作为当前 Mixtape 自动录制工程的起始节奏基准。

### 2. 所有歌曲初始状态先统一到项目 BPM

- 项目创建后，所有歌曲先以项目初始 BPM 作为默认目标 BPM
- 因此，初始状态下的全局 BPM 包络应表现为一条水平直线
- 用户看到的是一个已经成立的“项目节奏基线”，而不是各首歌原始 BPM 的拼接结果

这样定义后，“全局 BPM”才是一个真实存在、可以被编辑的对象。

### 3. BPM 包络是在项目 BPM 基线之上的时间变化

- BPM 包络表示项目在时间线上如何变速
- 用户修改包络，本质上是在修改整个项目在某一时间段的目标节奏
- 歌曲在该时间段内应服从这个项目节奏

用户编辑的不是某一首歌单独的速度曲线，而是整条 Mixtape 自动录制时间线的节奏走势。

### 4. Overlap 与 BPM 设计解耦

- 是否重叠，是时间编排问题
- 是否统一 BPM，是节奏策略问题
- 两者不应互相绑定

不建议通过“默认给所有歌曲加一段重叠”来制造一个看起来合理的全局 BPM。

更合理的方式是：

- 先定义项目 BPM
- 再决定是否生成过渡区

## 初始状态设计

用户首次进入 Mixtape 自动录制编辑界面时：

- 项目自动确定一个初始 BPM
- 所有歌曲默认先同步到这个 BPM
- BPM 包络默认是一条水平直线
- 时间线上存在统一的项目节奏参考

这个初始状态应该让用户一眼就明白：

“现在整个项目是按同一个速度在运行，我接下来可以通过包络去做整体节奏变化。”

## BPM 包络的交互定位

### 1. 它是项目级控制，不是轨道级控制

- BPM 包络应明确表现为 `Project / Master Tempo`
- 不应让用户误以为是在修改某一首歌的单独 BPM
- 它的作用范围是整个 Mixtape 自动录制工程

### 2. 它更接近“节奏编排”，而不是“参数涂抹”

- 用户添加的每个点，都代表项目节奏在某个时间位置上的锚点
- 用户拖动点时，关注的是“这一拍 / 这一小节开始变成多少 BPM”
- 整体交互应该像编排节奏段落，而不是像画音量曲线

### 3. 默认应服务于网格

- BPM 变化点默认对齐到网格线
- 用户编辑时应明显感知到当前正在对齐哪一条线
- 编辑节奏的基本单位应随缩放等级变化，例如小节、若干拍、单拍

默认行为应体现为：

- 先找节拍位置
- 再设置该位置的 BPM

而不是反过来按时间轴上的任意秒数落点。

## BPM 线的显示位置

### 1. BPM 线应位于时间尺下方、轨道区上方

- BPM 线属于整个项目，而不是某一条歌曲轨道
- 它应作为时间线中的一条独立 `Master Tempo Lane`
- 视觉层级应为：`时间尺 -> Master Tempo Lane -> 轨道区`

这个位置的优点是：

- 它与时间轴、播放头、网格和歌曲边界天然处于同一视野
- 用户在调整过渡区时，视线移动最短
- 不会被误认为是某一首歌自己的 BPM 曲线

### 2. 不建议的显示位置

- 不建议嵌入某一条歌曲轨道内部
- 不建议放到底部面板
- 不建议放在侧边栏
- 不建议做成独立弹窗或浮层编辑器

### 3. 视觉形态上应是时间线的一部分

- 不应再表现为一块悬浮的小卡片
- 应表现为时间线中的第一条主轨道
- 横向滚动、缩放和播放头应与主时间线同步

## Master Tempo Lane 的默认显示策略

### 1. 平时默认收起显示

- 非 BPM 编辑模式下，Master Tempo Lane 保持收起态
- 收起态的职责是持续提供项目节奏上下文，而不是承担完整编辑
- 用户不需要进入 BPM 模式，也能知道项目当前节奏状态

### 2. 进入 BPM 模式后展开

- 当用户切换到 BPM 编辑模式时，Master Tempo Lane 原地展开
- 展开后再显示网格、控制点和完整编辑能力
- 不建议切换到单独页面、弹窗或下方面板

这种方式可以兼顾：

- 平时不抢空间
- 编辑时上下文不丢失
- 视觉位置始终稳定

## 收起态建议显示的信息

收起态建议只保留少量高价值信息：

- `项目节奏 / Master Tempo` 标签
- 一条简化后的 BPM 走势线
- 当前播放位置对应的 BPM 数字
- 当前是否已经被编辑过的轻量状态提示

收起态不建议显示：

- 控制点
- 密集的 BPM 文本标签
- 过多按钮
- 与完整编辑有关的复杂操作

收起态应该更接近“节奏仪表”，而不是“编辑器”。

## 展开态建议高度

- 展开态建议高度控制在 `80 - 96px`
- 当前倾向使用约 `84px` 作为默认高度

这样做的原因是：

- 低于该区间，控制点、标签和网格会显得拥挤
- 高于该区间，会过度挤占主轨道区域的可视空间

展开态应承担完整编辑职责，因此允许：

- 显示网格线
- 显示控制点
- 显示拖拽反馈
- 在交互时显示当前点的 BPM 数字

## 播放头 BPM 的实时显示

建议实时显示当前播放头位置对应的 BPM 数字。

原因如下：

- 用户听到的是“当前速度”，界面也应同步告诉用户“当前速度是多少”
- 当项目存在升速、降速、过渡段时，实时数值可以显著降低理解成本
- 这条数值能让收起态依然具备明确的信息价值

建议表现方式：

- 收起态中持续显示当前播放位置的 BPM
- 展开态中也显示当前播放位置的 BPM
- 数值表现应轻量、稳定，不应喧宾夺主

## 当前推荐的显示方案

综合当前讨论，推荐采用如下方案：

- BPM 线固定在时间尺下方、轨道区上方
- 它是时间线内的独立 `Master Tempo Lane`
- 默认常驻，但以收起态显示
- 切换到 BPM 编辑模式后，原地展开
- 收起态显示项目节奏概览和当前 BPM
- 展开态显示完整的 BPM 编辑能力

## 建议的编辑行为

### 1. 初始包络

- 初始只有起点和终点两个点
- 两个点 BPM 相同
- 形成一条水平直线

### 2. 新增变化点

- 用户在 BPM 轨道上点击某条网格线附近时，插入一个新的 BPM 点
- 新点默认落在当前可见网格的吸附位置
- 新点的初始 BPM 可继承该位置当前的包络值

这样用户新增一个点时，不会立刻破坏曲线形状，只是先建立一个可编辑锚点。

### 3. 拖动变化点

- 水平方向：默认吸附到网格线
- 垂直方向：修改该点的目标 BPM
- 拖动时应强调当前将要落到的节拍位置

### 4. 删除变化点

- 起点和终点默认保留，不允许直接删除
- 中间点允许删除

### 5. BPM 数值表达

- 轨道上应让用户清楚看见当前点的 BPM 数值
- 用户应能快速理解当前变化是升速、降速还是保持

## 关于歌曲原始 BPM 的认知处理

虽然项目进入时会把所有歌曲先统一到项目 BPM，但用户仍然会关心每首歌原始 BPM。

因此界面认知上应区分：

- 原始 BPM：歌曲本身的分析结果
- 项目 BPM：当前 Mixtape 自动录制工程希望它播放到的目标节奏

建议在产品表达上避免混淆这两个概念。

## 关于过渡区的设计立场

本阶段不把“自动生成 overlap”作为 BPM 包络的前提条件。

建议产品语义保持为：

- 先建立统一项目 BPM
- 再决定是否为相邻歌曲生成标准过渡区

也就是说：

- `tempo` 先成立
- `transition` 再发生

这样用户心智更稳定，功能职责也更清晰。

## 预期用户收益

采用该交互定义后，用户会得到以下更稳定的体验：

- 一进项目就能看到一条有意义的 BPM 基线
- 不需要先处理每首歌原始 BPM 不一致的问题
- 可以把注意力集中在“哪里需要变速”
- BPM 变化点与网格对齐，更像在编排而不是在描线
- 过渡区和节奏策略不再互相干扰

## 当前讨论下的建议结论

建议采用以下默认规则：

- 项目初始 BPM 默认取第一首歌的 BPM
- 所有歌曲进入项目后先统一到该 BPM
- 初始 BPM 包络是一条平直的水平线
- 用户后续通过 BPM 包络修改项目节奏走势
- BPM 变化点默认对齐到网格线
- 是否生成歌曲间重叠，作为独立的时间编排策略处理

## 待继续讨论的问题

- 当第一首歌 BPM 不可靠时，项目是否需要提供更明显的初始 BPM 修正入口
- BPM 包络轨道是否需要提供“保持”“渐变”这类更明确的节奏段语义
- 用户是否需要一键生成标准节奏变化模板，例如升速段、降速段、过渡段
- 项目网格是否需要独立于某首歌曲存在明确的视觉表达
