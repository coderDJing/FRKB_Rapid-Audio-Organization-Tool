# 双轨和编辑模式大波形体验与架构标准

维护状态：长期维护

更新时间：2026-06-26

## 文档定位

这个文档是双轨模式和编辑模式大波形的长期维护标准，用于验收用户体验，也用于约束后续大波形相关改动的架构边界。

本文不是一次性问题复盘，也不是某次修复的交付说明。后续只要双轨模式或编辑模式的大波形体验、显示 owner、渲染链路、缓存链路、overlay、sync、seek、drag、zoom 或编辑能力发生实质变化，都应同步更新本文。

验收时只从用户能看到、能操作、能感受到的结果判断。只要用户体验不达标，即使技术上“已经显示了波形”，也不能算通过。

本文的验收口径以维护者手测看到的真实界面体验为准。只要维护者肉眼看到联结、拖拽、缩放、播放、起点前空白或 grid 对齐存在问题，本项就按不通过处理。

后续修改双轨或编辑模式大波形前，必须先判断改动属于哪个架构层、谁是当前操作的显示 owner、是否会影响声音优先级、是否会让波形和覆盖层分开更新。不能为了修一个局部问题，让播放、拖拽、缩放、seek、sync 或布局切换中的其它验收项回归。

维护本文时应区分三类内容：

- 长期不变量：声音优先级、波形清晰度、覆盖层同步、sync 对齐、播放头可信度、主题和 DPI 可读性。
- 当前架构状态：只描述当前 checkout 的真实路径和职责边界，过期后必须更新。
- 维护规则：约束后续改动的入口、owner 和回归线，不能替代真实代码审查和运行时验证。

## 术语边界

本文必须区分“联结”和“BeatSync”，禁止把二者混成同一种拖拽语义：

- 联结：指双轨联结按钮开启后的双 deck 视图联动。联结开启后，拖拽或缩放任意一条 deck 都等同于操作双轨整体，两条 deck 的 viewport、grid、cue、loop 和播放头关系必须同轮更新。
- BeatSync：指单条 deck 的 BeatSync 按钮。BeatSync 只表示当前 deck 与另一条 deck 做 tempo / beat-grid 对齐，不表示双轨 viewport 联动，也不表示拖拽时两条 deck 要一起平移或缩放。
- BeatSync 按钮只允许表达开 / 关两种状态；`tempo-only`、phase 未完全锁住或其它中间状态不能让按钮闪烁，也不能让用户误以为还有第三种 BeatSync 模式。
- 联结和 BeatSync 的 grid 对齐只表示可见 beat/grid 位置和播放相位进入对齐关系，不表示任何 deck 的大节线、小节线、phrase 或 `barBeatOffset` 定义要被另一条 deck 覆盖。如果主轨的大节线对齐的是从轨的小节线，从轨那根线仍必须保持小节线。
- BeatSync 开启后拖拽某一条 deck 时，拖动期间只允许当前被拖动的 deck 跟手移动；另一条 deck 应作为对齐参照保持自己的 viewport 和播放状态，不能像联结模式那样被拖着走。
- BeatSync 拖拽松手时，只能移动被拖动的 deck：被拖动 deck 必须以用户松手位置为 anchor，就近吸附到能与参照 deck grid 对齐的位置。参照 deck 不能因为这次 BeatSync 拖拽被二次 seek、二次吸附或整体平移。
- 如果被拖动 deck 原本是 BeatSync leader，松手吸附时应以另一条静止 deck 为参照，使被拖动 deck 对齐过去；不能为了保持原 leader 身份而反向移动另一条 deck。
- 如果另一条静止 deck 原本是 BeatSync follower，被拖动 leader 松手时，静止 deck 必须退出 BeatSync 并接管 MASTER，继续按自己的播放状态正常播放；不能一边显示为 MASTER 一边继续执行 BeatSync follower 逻辑。
- 如果某条 deck 处于暂停 / 未播放状态，另一条 deck 正在播放，此时点击暂停 deck 的 BeatSync 只能进入 tempo-only：允许暂停 deck 的 BPM / playbackRate 和 grid 间距变化，但不能为了追正在播放的 deck 修改暂停 deck 的 currentSec、播放头、viewport 或 stable frame 锚点。
- 如果某条 deck 已经开启 BeatSync、处于暂停状态，用户点击播放时，播放开始前必须完成对参照 deck 的 beat/grid 吸附；不能先按暂停时的旧 currentSec 启动声音或渲染时钟，再等下一轮 native snapshot / render sync 才追到网格线上。
- 本文后续凡写“联结后整体拖拽 / 整体缩放”，只适用于联结按钮语义，不适用于 BeatSync。凡涉及 BeatSync 的单轨拖拽、松手吸附、网格对齐，必须明确写 BeatSync。

近期已经明确暴露出来、必须长期纳入验收的问题包括：

- 载入或换歌首帧，歌曲起点前出现密集 grid、节拍刻度或小节线。
- 点击联结或 BeatSync 后，上下 deck 网格线仍然歪，必须再拖拽一下波形才真正对齐。
- 播放中点击联结或 BeatSync 后，上下 deck 网格线没有立即对齐，或者播放滚动中一轨先跑、另一轨随后追上。
- 联结或 BeatSync 开启瞬间先抽动、先缩放、先偏移，随后才贴合。
- 开启联结瞬间出现可见卡顿，或者第一次开启时从轨播放进度先跳到奇怪位置再跳回。
- 开启联结或 BeatSync 瞬间出现波形缺块、旧帧残留、波形和声音对不上。
- 点击 MASTER / leader deck 的 BeatSync 时，声音没有移动，但主轨大波形自己右移，造成波形和声音错位。
- 点击联结或从轨 BeatSync 后，从轨原本对齐到主轨大节线的小节线被错误显示成大节线。
- 联结后拖拽任意一条 deck 时，两条 deck 没有像一个整体一样平移。
- BeatSync 后拖拽任意一条 deck 时，另一条 deck 被错误地跟随平移，或松手后当前 deck 没有按松手位置就近吸附到对齐 grid。
- BeatSync 拖拽松手后，transport / beat phase 已经对齐，但屏幕上的大波形继续复用松手前旧 stable frame，导致 grid 线肉眼可见偏移一到数个像素。
- 联结或 BeatSync 后拖拽松手时，某一轨 grid 线层级短暂重判，出现大节线、小节线、普通线来回切换或像闪烁一样几帧后才恢复。
- 一条 deck 正在播放、另一条 deck 暂停时，点击暂停 deck 的 BeatSync 后暂停 deck 波形追向播放 deck，或播放头 / viewport 因 BPM 变化发生位移。
- 从轨 / follower 已开启 BeatSync 且处于暂停状态时，点击播放后偶发没有吸附到参照 deck 网格线，或先按旧位置启动再追到网格线。
- 联结后滚轮缩放任意一条 deck 时，两条 deck 没有像一个整体一样缩放。
- 已有波形数据的歌曲在播放、拖拽、seek 或缩放时仍表现得像现场计算，出现闪烁、热气感、重绘抖动或卡顿。

## 架构边界

本节约束的是大波形相关代码的方向盘归属。这里的目标不是描述每个函数怎么写，而是防止多个层同时修改同一个视觉结果。

### 相关 Surface

FRKB 里和波形相关的 surface 至少包括：

- 歌曲列表波形预览。
- 浏览器模式主播放器波形。
- 双轨模式概览波形。
- 编辑模式概览波形。
- 双轨模式 detail 大波形。
- 编辑模式 detail 大波形。

这些 surface 可以复用缓存、颜色、渲染工具或视觉语言，但不能默认共用运行时状态机。任何跨 surface 复用都必须先说明复用的是数据、样式、渲染函数还是交互状态，禁止把一个 surface 的播放、seek、拖拽、缩放状态直接打进另一个 surface。

### 分层事实源

大波形相关代码应按以下事实源分层：

- 音频事实源：native transport 负责声音播放、暂停、seek、同步倍率、leader / follower、playhead 加载状态。声音优先级最高，任何波形显示准备都不能阻塞声音。
- 歌曲时间线事实源：song grid、BPM、first beat、bar beat offset、time basis、cue、loop 是时间线事实源。它们决定某个音乐事件在歌曲时间上的位置。
- 波形数据事实源：已分析波形缓存、统一 display waveform、compact visual、detail waveform 等只描述可绘制的波形形状。它们不是播放事实源，也不是当前 viewport 的 owner。
- 视口和 presentation 事实源：当前用户关注哪一段、播放头显示在哪里、可视窗口宽度和缩放比例，由当前交互 owner 决定。
- 覆盖层事实源：grid、cue、loop、播放头必须从同一个时间线和同一个 viewport 映射得到，不能和波形本体各算一套屏幕坐标。grid 可以在计算层作为虚拟时间线向歌曲 0 秒之前延伸，用于保证双轨联结时完整节拍尺对齐；但实际覆盖层内容必须裁剪到歌曲真实时间线内，不能在起点前画出 grid、刻度、cue、loop 或假波形。这个约束同样适用于首次载入、换歌、缓存恢复、worker 首帧返回、sync / 联结事务首帧和空态占位。
- 主题和 DPI 事实源：主题、颜色、线宽、device pixel ratio 只能影响视觉样式和清晰度，不能改变歌曲时间、播放位置或对齐关系。

### 单一 Owner 规则

同一时刻只能有一个 owner 负责提交大波形的可见位置和 presentation。其它层可以提供数据、快照或候选结果，但不能抢着提交最终画面。

- 普通播放时，稳定 presentation / canvas 播放循环是显示 owner；普通 currentSeconds watcher 只能校准，不应每帧重新抢 viewport。
- seek 时，seek intent 是显示 owner；近距离 seek 应复用当前稳定画面，远距离 seek 可以请求新帧或短暂受控空态，但不能新旧位置来回跳。
- 拖拽时，pointer / scrub 交互是显示 owner；native snapshot 或播放状态回传不能在拖拽中把 viewport 拉回旧位置。
- 联结开启时，shell / fader 层的一次性视觉事务是显示 owner；上下 deck 的 grid、视觉节拍尺度、viewport anchor 必须同一轮提交。不同 BPM 的歌曲可以通过 tempo / time scale 补偿得到同一屏幕节拍间距，但 detail lane 不能先各自进入半成品 linked 状态。
- BeatSync 开启时，BeatSync 对齐事务是显示 owner；它必须让当前 deck 和参照 deck 的 grid 进入对齐关系，但不能把两个 deck 误切成联结拖拽或联结缩放状态。
- BeatSync 开启时，显示事务只能移动或重锚确实被 native transport 移动的 deck。native 没有移动 leader / MASTER deck 时，renderer 不能为了“看起来同步”单独右移主轨波形。
- BeatSync 开启时，如果目标 deck 没有播放、另一条 deck 正在播放，native 和 renderer 都只能走 tempo-only 语义：native 可以更新 syncEnabled、leader、syncLock、playbackRate / BPM multiplier，但不能执行 phase alignment；renderer 可以重算 grid 间距和播放倍率，但不能创建 linked / BeatSync visual transaction 去重锚暂停 deck。
- 联结或 BeatSync 开启时，可以为了对齐更新 viewport、playback clock、time scale 或 beat 位置，但不能重定义任何歌曲时间线事实源：`firstBeatMs`、`barBeatOffset`、大节线/小节线层级和 phrase 语义仍属于各自 deck 自己的 grid。
- 联结后拖拽时，整体平移事务是显示 owner；拖动一条 deck 等同于拖动双轨整体，另一条 deck 不能停住等松手。
- BeatSync 后拖拽时，当前 deck 的 pointer / scrub 交互是显示 owner；另一条 deck 是对齐参照，不是被拖拽对象。
- 联结后缩放时，共享 zoom 状态是显示 owner；两条 deck 必须同一轮更新可视时间尺度和关注位置。
- 联结拖拽松手到 native commit 完成之间，拖拽释放事务仍是显示 owner；gridTimeBasis、bar phase 和线条层级不能短暂回到某条 deck 的原始歌曲 grid，也不能重新动态推导 follower 的 barBeatOffset，再由下一帧 linked phase 或稳定帧修正。
- BeatSync 拖拽松手到 native align 完成之间，当前 deck 的拖拽释放事务仍是显示 owner；当前 deck 可以按松手位置吸附到参照 deck grid，但参照 deck 不能被二次移动，且释放过程不能出现明显等待、卡顿或旧位置残留。
- BeatSync 拖拽松手后的 release 视口必须跟随吸附后的当前 deck 播放点重新锚定；不能继续使用鼠标松手瞬间的旧 viewport 或旧 stable frame 来证明“画面已稳定”。如果 native align 把 anchor 吸附到新位置，显示层必须以吸附后的 anchor 作为最终画面事实源。
- 编辑 grid、BPM、cue、loop 时，对应编辑交互是事实源；波形本体和覆盖层必须同一时间线更新，不能一个先变、另一个后追。
- 首次载入、换歌、缓存恢复或 worker 首帧返回时，加载 presentation 是显示 owner；它只能决定显示稳定帧、受控空态或真实波形，不能借机外推 0 秒前 grid，也不能把未裁剪的临时 overlay 暴露给用户。
- worker 或缓存加载完成时，worker 只能提交“某段波形数据 ready”，不能自行改变用户当前关注位置。

### 禁止抢方向盘的模式

以下模式即使短期看起来能修某个 bug，也不应作为最终方案：

- 在播放、拖拽、seek、缩放或 sync 事务中，子组件 watcher 用旧 currentSeconds 重新计算并覆盖 previewStartSec。
- worker frame ready 后，直接按旧请求的 anchor 覆盖当前 viewport。
- grid、cue、loop、播放头和波形本体分别使用不同的 viewport、time basis 或缩放倍率。
- 已分析歌曲在播放、拖拽、seek 或缩放中重新走现场 raw stream 状态机，导致用户感觉还在重新计算波形。
- native snapshot 在拖拽、缩放、seek 或联结视觉事务中提前打穿显示层，造成先动一下再贴合。
- 为了避免空态而显示旧位置波形，导致用户误判当前位置。
- 加载态、空态或未拿到稳定 viewport 时，先按默认 grid 间距铺满画布，再等真实波形或真实 viewport 返回后修正。
- 为了让联结或 BeatSync 看起来对齐，在歌曲起点前反推、复制、补画或密集铺设 grid 线。
- 为了让联结或 BeatSync 看起来对齐，重算 follower 的 `barBeatOffset`，把从轨小节线临时或长期改成大节线。
- 为了让波形先稳定而暂停、延迟、重启或拖慢声音。
- 只修当前 deck，不检查另一条 deck、overview、overlay、theme、DPI 和编辑模式是否被带坏。

### 修改前检查清单

任何触及双轨或编辑模式大波形的改动，至少先回答这些问题：

- 这次改的是哪个 surface：列表预览、主播放器、双轨 overview、编辑 overview、双轨 detail，还是编辑 detail？
- 这次改的是哪个层：音频 transport、歌曲时间线、波形数据、viewport / presentation、overlay，还是主题 / DPI？
- 当前用户操作的显示 owner 是谁：播放、seek、拖拽、缩放、联结事务、grid 编辑，还是加载态？
- 这次改动是否会让两个 owner 同时提交 viewport？
- 这次改动是否会让波形本体和 grid / cue / loop / 播放头分开更新？
- 这次改动是否会让 native snapshot 或 worker ready 结果覆盖用户当前关注位置？
- 这次改动是否会影响声音连续性、播放头可信度、联结 / BeatSync 对齐、联结整体拖拽、联结整体缩放或编辑模式深度 zoom？

### 当前架构状态

截至 2026-06-22，双轨和编辑模式 detail 大波形已经采用显式 presentation owner 架构。当前主链路是：

`用户操作 intent / transport 校准 / waveform data ready` -> `HorizontalBrowseModeShell.vue` -> `WaveformPresentationCoordinator` -> `detail lane presentationState` -> `HorizontalBrowseRawWaveformDetail.vue` -> `canvas renderer / worker`。

这里的核心边界是：shell 负责跨 deck 事务入口，coordinator 负责 owner、revision 和双轨 presentation state，detail lane 负责透传和局部交互，raw waveform detail 负责消费 state 并发出 pointer / wheel / grid edit intent，canvas / worker 负责按 state 绘制。

当前 `HorizontalBrowseWaveformPresentationState` 至少包含 `owner`、`revision`、`sourceDeck`、`affectedDecks`、`anchorSec`、`viewportStartSec`、`visibleDurationSec`、`anchorRatio`、`zoom`、`timeScale`、`gridTimeBasis`、`playbackClock`、`surfaceMode`、`linked`、`visualPending`、`updatedAtMs`。这些字段共同描述某一轮画面提交，后续新增视觉事务必须优先进入这个模型。

### 当前模块职责

- `HorizontalBrowseModeShell.vue`：组合双轨 surface，接入高层播放、seek、drag、zoom、sync 和 stable frame 事务，是 detail presentation 的上层入口。
- `horizontalBrowseWaveformPresentationCoordinator.ts`：维护 top / bottom 的 presentation state，统一提交 owner、revision、source deck、affected decks、viewport、zoom、time scale、surface mode、linked 和 visual pending。
- `horizontalBrowseWaveformPresentationShellBridge.ts`：把 shell 层的拖拽和缩放事件转换为 coordinator transaction；联结拖拽和联结缩放必须作为双 deck transaction 提交。
- `horizontalBrowseModeShellDetailTransactions.ts`：承接 sync visual transaction、stable frame 准备和 detail lane 的视觉事务桥接，避免两条 lane 各自进入半成品状态。
- `HorizontalBrowseDeckDetailLane.vue`：作为 detail lane 包装层，透传 stable frame、linked visual commit 和 presentation state，不拥有跨 deck presentation 决策权。
- `HorizontalBrowseRawWaveformDetail.vue`：消费 presentation state，维护 detail 本地网格和交互入口，可以发出 pointer、wheel、seek、grid edit intent，但不能绕过 coordinator 决定双轨最终 viewport。
- `horizontalBrowseDetailPresentationConsumer.ts`：按 coordinator revision 应用 seek、zoom、viewport 和 anchor，同时保留 shared zoom 兼容路径。
- `horizontalBrowseDetailPlaybackPositionWatch.ts`：处理 playback / currentSeconds 校准、稳定播放帧复用和 seek handoff；当 presentation transaction 或 visual pending 存在时不能抢回 viewport owner。
- `horizontalBrowseDetailPresentationState.ts`、`horizontalBrowseDetailPresentationActions.ts`、`horizontalBrowseDetailGridPersistence.ts`：维护 detail 局部 presentation、稳定帧准备、time scale 应用和 grid 持久化。
- `useHorizontalBrowseRawWaveformCanvas.ts` 与 `horizontalBrowseRawWaveformViewport.ts`：负责 canvas renderer、frame cache、可见性、worker bridge 和 viewport 计算工具，不能成为跨 deck presentation owner。
- `useHorizontalBrowseDeckPlaybackController.ts` 与 `horizontalBrowseLinkedRawWaveformDragEnd.ts`：负责 transport、scrub、seek intent 和 linked drag release native commit，不直接决定最终 detail viewport。
- compact / unified display waveform 相关模块：提供已分析歌曲的显示域波形数据；它们是 waveform data provider，不是播放或 presentation owner。

### 当前 Owner 规则落地

- 播放：稳定 presentation / canvas 播放循环拥有可见移动；`currentSeconds` watcher 只做校准，不能每帧覆盖 coordinator state。
- seek：shell 标记 `owner = 'seek'` 并提交 revision；detail consumer 按 revision 应用目标位置，旧 seek watcher 只作为无 coordinator 场景的兼容路径。
- 拖拽：pointer / scrub 事务拥有画面；联结拖拽以同一 revision 同时提交 top / bottom，另一条 deck 不能靠 watcher 事后追赶。
- 缩放：wheel intent 提交 zoom、anchor 和 viewport；联结缩放以 `owner = 'linked-zoom'` 同轮更新两条 deck 的可视时间尺度和关注位置。
- 播放中缩放：当前播放头位置是缩放 anchor；如果 presentation 携带的 `playbackClock` 与当前播放位置漂移超过容忍阈值，应丢弃旧 clock 并用真实当前播放位置重锚，不能把旧 clock 继续用于 zoom viewport。
- 联结缩放渲染：`zoom` / `linked-zoom` 是视口和可视时间尺度变化，不是 stable frame 身份切换；连续滚轮缩放不能每一跳都清空 stable presentation、停止 stable playback 或强制重置 display ready。需要新帧时可以按当前 viewport 重绘或重锚，但不能让旧帧、新帧、overlay 在连续 wheel 事件中来回抢显示。
- 联结或 BeatSync 开启：shell / fader 以 `owner = 'sync-transaction'` 包裹 visual transaction，`visualPending` 阻止半成品对齐状态被渲染或被旧 snapshot 覆盖。
- 开启联结或 BeatSync 时，visual transaction 必须先于 native sync / align 的可见状态变化进入 pending；不能等 native 已经改完 tempo、leader、syncLock 或播放位置后，再让 detail lane 事后追赶新关系。
- 联结开关、BeatSync 开关、拖拽松手、seek release、sync release 这类从一个显示 owner 切到另一个 owner 的操作，都必须按 presentation handoff 处理。交接期间如果旧 stable frame 的 grid、viewport、playback clock 仍然语义正确，可以继续用旧帧滚动；如果旧帧已经不正确，必须先准备或重锚新 stable frame 再可见切换，不能让 detail lane 进入没有有效 stable frame、没有 playback RAF 或只能等待 worker 首帧的停顿状态。
- 联结关闭、BeatSync 关闭、开启联结 / BeatSync 和拖拽松手如果需要新 worker frame，旧 stable presentation 必须作为过渡画面保留到新帧 ready；不能先清空 stable presentation、重置 displayReady 或隐藏 surface，再等 worker 回来补画，否则用户会看到卡顿、黑一下或旧帧停住。
- grid、BPM、cue、loop 编辑：编辑交互是时间线事实源；波形本体、overlay、播放头必须使用同一 viewport / time basis 派生。
- worker / cache ready：只能提升某个 revision 的可绘制帧，不能反向改变用户当前关注位置。

### 维护规则

- 新增或修改 seek、drag、zoom、BeatSync、linked、grid edit 相关行为时，默认从 `HorizontalBrowseModeShell.vue` 或对应 shell bridge 进入 coordinator；只有联结语义可以提交跨 deck viewport，BeatSync 拖拽只能提交当前 deck 对齐。
- 如果确实需要新增 owner，必须同时更新 `HorizontalBrowseWaveformPresentationOwner`、state 字段、shell transaction、detail consumer 和本文档。
- overlay、waveform body、grid、cue、loop、playhead 必须使用同一个 viewport / time basis / zoom 来源。只改其中一条路径不算完成。
- 联结 / BeatSync 对齐必须通过同一轮 presentation transaction、time basis、bar phase、playback clock 和 viewport 映射解决；禁止用“在歌曲起点前外推、反推或补画大量 grid 线”来伪装对齐。
- native transport snapshot 是声音和校准输入，不是用户交互中的显示 owner。用户 intent 进行中时，snapshot 不能覆盖 intent 指定的 viewport。
- canvas 和 worker 可以管理旧帧、空态、可见性、token 和 frame ready，但不能决定用户关注哪一段。
- 双轨联结下的拖拽、缩放、联结开关都必须以 top / bottom 同一轮 revision 提交，不能恢复为“源 deck 先动，另一条 deck 后追”的模式。BeatSync 不享有联结拖拽语义；BeatSync 松手只允许被拖动 deck 按松手 anchor 对齐参照 deck。
- BeatSync 拖拽松手不能先把未吸附的鼠标松手点提交成最终 seek 画面，再等 native align 后二次跳到吸附点；显示层必须把拖拽释放画面保留到 native 返回吸附后的 deck 秒数，再以吸附后位置作为最终 presentation 事实源。
- `stableRenderRevision` 这类用于判定稳定画面身份的机制，只能由真正改变稳定参考关系的 owner 驱动，例如 seek、linked playback、drag 或 linked drag。普通 zoom / linked zoom 不能纳入强失效条件，否则连续滚轮会反复清空稳定帧，造成波形闪烁、波形和 grid 分离或上下 deck 像在各自切画面。linked playback 这类需要新 revision 帧的播放中 handoff，必须允许旧 stable frame 按自身 viewport 时钟继续滚动到替代帧 ready；禁止把新 linked viewport 直接套到旧 frame 上，导致一帧 `presentable=false` 后断掉 playback RAF。
- `visualPending` 只能用来挡住错误的半成品画面，不能把它当成可感知冻结的借口。native、worker 或双 lane commit 还没完成时，显示层仍应保持一个语义正确的可展示画面继续滚动，或者使用明确受控的短过渡；不能因为等待事务完成而让播放中的波形肉眼停住一下。
- `preserveDisplay` 必须真的保留当前 stable presentation、rendered range 和双缓冲可见面，直到替代帧可展示。禁止只保留 DOM opacity，却把 stable canvas playback、presentation frame 或 revision handoff 清空。
- 手测是本文唯一的验收通过口径。内部状态或工具输出最多只能帮助定位问题，不能把维护者肉眼看到的错位、闪烁、抽动、卡顿或起点前假内容改判为通过。
- 修改本文验收标准前必须先和维护者说明并确认。说明内容至少包括：为什么当前标准需要改、改动会放宽还是收紧、影响哪些验收项、原本失败的场景按新口径会如何判定。禁止把标准变更混在功能修复里偷偷落地。

2026-06-26 复测失败记录：尝试在 stable revision 变化时继续保持 `displayReady = true`，确实命中了 `[HB-LINK-OPEN] canvas-preserve-display-ready-stable-revision`，但用户看到开启联结瞬间波形部分缺失、仍然卡顿，并且多次开关后出现波形和声音对不上的严重回归。这个方向不能作为候选修法继续保留；“保可见面”不能等价于“保 displayReady=true”，否则旧 stable frame、pending linked frame、overlay 和 playback watcher 会同时认为自己仍可提交可见状态。

同一轮复查还发现关闭联结链路是 fire-and-forget：renderer 先清 linked presentation，但 `nativeTransport.setSyncEnabled(false)` 没有等待完成，快速多次开关时下一次开启可能和上一轮关闭的 native snapshot 交错。这个竞态能解释“多次开启关闭后波形和声音对不上”的状态错位风险，关闭链路必须串行化并记录落盘日志后再允许下一次开启。

2026-06-26 继续复测后，波形缺块和多次开关后的视听错位已经消失，关闭联结不再明显卡顿，但开启联结瞬间卡顿仍然存在。维护者进一步观察到第一次开启联结时，波形进度会先瞬间移动到奇怪位置，再瞬间移动回来。新日志显示剩余卡顿仍集中在 `canvas-reset-display-ready-stable-revision` 到新 `stable-handle-rendered` 之间，通常为数十毫秒；同时在 `sync-transaction` pending 期间，即使已经取消后续 tick 的反复 reanchor，`playback-watch-pending` 仍会把连续变化的半成品 `currentSeconds` 通过 `applyStableCanvasPresentation` 写进 stable presentation。第一次开启时 follower deck 还会在最终 `linked-playback` commit 前提前触发一次 `stable-start-playback`。pending 期间旧 stable frame 应该按自己的 RAF 时钟继续滚动，不能让 currentSeconds watcher 或 playbackActive watcher 抢走联结事务的显示 owner。

2026-06-26 再次复测后，上述 watcher 冻结已经生效，日志里 `playback-watch-pending` 变为 `applied:false / presentationFrozen:true`，但开启仍卡，且第一次开启时从轨进度仍会后跳再回来。新证据指向更深的架构问题：`currentSeconds` 同时承担“可见渲染时间”和“最终 native 事务提交时间”两种职责。开启联结时，从轨会在 pending 阶段收到 native `setSyncEnabled` / `alignToLeader` 产生的 BPM / playbackRate / sync phase 中间态，例如从 `playbackRate:1` 变成 `0.968254`，随后 `snapshotSec` 从 `0.026` 跳到 `0.197`，并触发 `signatureChanged`、`fullSyncPhaseChanged` 和 playback revision bump。只冻结 detail watcher 不够，因为 render sync 仍然能把这些中间态写进 `topDeckRenderCurrentSeconds / bottomDeckRenderCurrentSeconds`，而 detail lane commit 又会在 pending 期间提前修改 `lastAppliedPreviewTimeScale`、`previewStartSec` 和 visual grid。也就是说，文档里的 owner 规则是对的，问题在于实现没有把音频事实源、可见渲染时间和最终 presentation commit 输入拆开，导致 `visualPending` 没有真正覆盖所有能改可见位置和时间尺度的入口。

2026-06-26 最新复测日志进一步确认：`render-sync-deck` 已经在 `visualPending:true` 时记录 `suppressedByVisualPending:true`，`detail-commit-linked-transaction` 也为 `mutatedLocalPresentation:false`，说明 render sync、currentSeconds watcher 和 pending lane commit 已经不再直接抢 owner。第一次开启仍出现从轨跳动的直接证据来自 stable handoff：从轨旧 `currentFrameRevision:0` 并非播放态画面，却被 `stable-tick-handoff-applied` 当作可播放旧帧，用旧 frame 的 `anchorStartedAtMs` 外推到约 `3.02s`；而同一轮最终 `presentation-finish-sync` 的从轨 native anchor 只有约 `0.105s`。这说明 stable handoff 不能只判断“有旧 frame + 有新 pending frame”，还必须知道旧 frame 生成时是否处于播放态。后续第一次开启跳怪位置消失，但开启瞬间仍卡的日志显示，简单禁止非播放态旧帧 handoff 会让从轨在等待 worker 新帧 ready 的 20-40ms 内停住。因此候选规则调整为：播放态旧帧按旧帧时钟继续滚；非播放态旧帧只有在旧 anchor 和新 pending anchor 很近时，才能用新的 playback clock 做短暂 static-to-playback handoff，不能再使用旧 frame 的创建时间外推。

2026-06-26 再次复测后，第一次开启跳到怪位置的问题已消失，但开启联结瞬间仍必定卡一下；当前 `log.txt` 没有拿到新的复现日志，不能把上一个候选写成已修复。新的代码侧候选把问题继续收窄到 `useHorizontalBrowseRawWaveformCanvas.ts` 的 stable revision replacement：`displayReady=false` 仍然表示“新 revision 尚未 ready”，但 DOM surface 可见性、旧 stable frame 是否可继续滚动、同 revision 替代帧是否已经在 worker 中渲染，必须拆成三个状态。尤其不能再用 `displayReady=true` 保旧面；而是用独立的 handoff surface 标记保护旧帧滚动，并跳过同一 stable revision 的低价值重复 render，避免连续 watcher 把 pending token 反复顶掉、延长用户看到的停顿窗。该候选仍需维护者复测验证。

2026-06-26 复测后仍能感到开启瞬间轻微卡顿，但日志显示问题已经进入更窄的成本区间：复现窗口内没有 `stable-tick-stop`，`stable-tick-handoff-applied` 持续出现，说明旧 stable frame 的播放 RAF 没有断；重复 render 也已经被 `canvas-skip-duplicate-stable-revision-render` 拦住。多数开启中上下 deck 的 `handoffPresentable` 为 `true`，但仍需要等待新 linked stable frame 首帧 ready，`canvas-reset-display-ready-stable-revision` 到 `canvas-stable-replacement-rendered` 常见约 18-34ms。继续优化不应再做 watcher 级小补丁，下一步要么接受这个轻微首帧成本，要么进入更重的架构方向：预构建 linked replacement frame、把 worker stable frame 渲染做成可抢占/增量、或让 linked 开启前就持有可直接切换的双 deck 对齐 frame。上述方向成本和回归风险明显高于当前窄修。

本轮代码侧候选修法：

- `useHorizontalBrowseRawWaveformCanvas.ts` 撤回“stable revision 变化时保持 `displayReady=true`”的尝试，继续保持 `displayReady=false` 作为“新 revision 尚未 ready”的事实源。后续如果要修这条链路，必须拆分“surface 可见性”和“当前 revision 可作为事实源”两个概念，不能再直接复用 `displayReady=true`。
- `useHorizontalBrowseRawWaveformCanvas.ts` 最新候选把 stable revision replacement 拆成三类状态：`displayReady` 只表示新 revision 是否 ready；`preserveSurfaceUntilNextReady` 只表示 DOM surface 是否继续可见；`stableRevisionHandoffSurfaceActive` 只表示旧 stable frame 是否可作为短暂滚动 handoff。替代帧 pending 期间，如果同一个 stable revision 已经有 worker render 在飞，后续非 anchor 的重复 draw 会直接跳过，避免 token 被连续重置、延长用户看到的停顿窗。
- `useHorizontalBrowseFaderControls.ts` 关闭联结时等待两条 deck 的 native sync 关闭完成，再 `syncDeckRenderState()`，最后按关闭后的 snapshot 清 linked presentation；关闭期间用 `dualTransportSyncDeactivating` 阻止下一次开启抢跑。
- `horizontalBrowseDetailPlaybackPositionWatch.ts` 在 `linkedGridVisualPending` 期间完全冻结 currentSeconds 对 stable presentation 的写入；pending 期间不再调用 `applyStableCanvasPresentation` 或 `reanchorStableCanvasPlayback`。最终 linked 位置只能由 `presentation-finish-sync` 后的 `linked-playback` revision 提交。
- `HorizontalBrowseRawWaveformDetail.vue` 在 `linkedGridVisualPending` 期间拦截 `waveformPlaybackActive` watcher；pending 期间只同步 `previewPlaying`，不能提前 `startStableCanvasPlayback`、`stopStableCanvasPlayback`、刷新 stable frame 或按半成品 currentSeconds 重锚。首次开启时 follower deck 的播放画面必须等最终 `linked-playback` commit 统一接管。
- `useHorizontalBrowseRenderSync.ts` 在 `WaveformPresentationCoordinator` 处于 `visualPending` 时抑制 native 中间 snapshot 对可见 render clock 的 reanchor 和 playback revision bump。pending 期间 render sync 可以继续按旧 base 平滑外推，但不能消费 `setSyncEnabled` / `alignToLeader` 造成的半成品 `playbackRate`、`snapshotSec` 或 `syncLock` 变化。
- `useHorizontalBrowseFaderControls.ts`、`horizontalBrowseModeShellDetailTransactions.ts` 和 detail lane expose 新增显式 native deck state 传递。最终 `linked-playback` commit 使用 native align 后的 `currentSeconds / playbackRate / playbackActive` 作为事务输入，而不是复用已被冻结的可见 render currentSeconds。这样可见 render clock 能在 pending 中保持稳定，最终 commit 仍能落到 native 已对齐的位置。
- `horizontalBrowseDetailPresentationActions.ts` 在 pending 的 `commitLinkedGridVisualTransaction` 中改成纯计算 result，不再提前修改本地 `previewStartSec`、`lastAppliedPreviewTimeScale` 或 visual grid；`mutatedLocalPresentation:false` 表示该 lane 没有在 coordinator `finishSyncTransaction` 前抢先改变画面。最终可见修改仍由 `linked-playback` presentation consumer 统一应用。
- `horizontalBrowseStableCanvasPresentation.ts` 在 revision handoff 期间识别“旧 current frame + 新 pending frame”的状态；只要替代帧还没 ready，`tickPlayback` 就用旧 frame 自己的 `viewportRangeStartSec / anchorSec / playbackRate` 继续推进画面，而不是用新 linked viewport 去测旧 frame 是否 presentable。这样即使新 linked revision 的 worker 帧还在后台渲染，旧 stable frame 也能继续滚动，不会因为一次 `presentable === false` 把 playback RAF 断掉。最新补充：旧 frame 必须携带 `playbackActive`。生成时处于播放态的旧 frame 可以按旧帧时钟 handoff；第一次开启从轨这类非播放态旧 frame 只有在旧 anchor 接近新 pending anchor 时，才能用新的 playback clock 做短 handoff，禁止按旧 frame 创建时间外推，否则会先跳到错误进度再被最终 linked commit 拉回。
- `useHorizontalBrowseFaderControls.ts` 开启联结后不再无条件 `syncDeckRenderState({ force: 'all' })`。force 模式会把两条 deck 的 render clock 强制重锚并 bump `playbackSyncRevision`，即使刚关闭再开启、grid / BPM / 速度关系没有可见变化，也会制造一次无意义的渲染时钟跳变。开启联结后应只做普通 `syncDeckRenderState()`，由真实 native snapshot 差异决定是否重锚。
- `HorizontalBrowseModeShell.vue` 不再把 `dualTransportSyncActivating` 直接传给 detail lane 当 `linkedGridVisualPending`。按钮 loading 只属于按钮状态，不能继续压住大波形的 pending 分支；大波形是否 pending 必须由 `WaveformPresentationCoordinator` 的 `sync-transaction / visualPending` 决定。否则 `finishSyncTransaction` 已经完成后，detail lane 仍会因为 UI hold 多停一帧。
- 2026-06-26 新增 BeatSync 主轨错位候选修法：BeatSync 开启不能继续复用完整联结开关的双轨 presentation 提交流程。点击 MASTER / leader deck 的 BeatSync 时，native `alignToLeader` 本身不会移动音频，renderer 也不能创建 linked visual transaction，否则会出现“声音没动、主轨波形右移”的视听错位。非 leader deck 开启 BeatSync 时，可以继续让两条 lane 参与计算对齐结果，但 leader lane 只能 compute-only，禁止修改本地 `previewStartSec`、timeScale 或 stable frame；coordinator 只允许把最终 `linked-playback` presentation commit 到 follower deck。leader deck 必须保持自己的 playback owner 和 stable frame，不得被 `force:'all'` 或 linked commit 连带重锚。
- 2026-06-26 新增联结 / BeatSync 网格定义候选修法：联结和 BeatSync 都只能让可见 beat/grid 位置与播放关系对齐，不能让 follower 的 `barBeatOffset` 和线条层级被 leader bar phase 覆盖。`horizontalBrowseModeShellDetailTransactions.ts` 只能对齐 playback clock，不能调用会重写 follower `barBeatOffset` 的结果对齐；绘制层的 linked phase 也必须始终保留当前 deck 自己的 bar 定义，禁止把“主轨大节线对齐从轨小节线”画成“从轨小节线变成大节线”。
- 2026-06-26 新增 BeatSync 拖拽松手候选修法：BeatSync 开启后拖动从轨或当前 sync follower 时，release 不能只调用 native `alignToLeader` 后靠普通 `syncDeckRenderState` 追画面。松手到吸附完成之间应由 BeatSync release visual transaction 持有显示 owner，native 返回吸附后的当前 deck 秒数后，再把该 deck 作为 follower 提交 `linked-playback` presentation；参照 deck 只能 compute-only，不能被重锚、二次 seek 或改写 grid 定义。release pending 只允许覆盖 native 吸附这段必要时间，不能再把 stable-frame 准备或强制 render sync 当成最终 presentation 提交前置条件，否则会把松手卡顿做实，也可能让声音 / transport 已经对齐但从轨大波形和 overlay 仍停在松手旧 viewport，最终肉眼看起来对不齐。
- 2026-06-26 新增 BeatSync leader 拖拽角色切换候选修法：当只有 follower 开着 BeatSync、leader / MASTER deck 没开 BeatSync，而用户拖拽 leader 时，release 必须先让原 follower 退出 BeatSync，再显式把原 follower 设为 MASTER / leader，然后只把被拖动 deck 对齐过去。这样静止 deck 可以继续正常播放，不会在成为 MASTER 后仍保留 `syncEnabled` / `tempo-only` / `full` 的 follower 状态。
- 2026-06-26 新增暂停 deck 开启 BeatSync 修法：当一条 deck 正在播放、另一条 deck 暂停 / 未播放时，点击暂停 deck 的 BeatSync 不能调用 native `alignToLeader`，也不能开启 BeatSync visual transaction。正确路径是只调用 `setSyncEnabled(deck, true)`，让 native `refresh_sync_state(false)` 更新倍率和 sync 状态，然后 renderer 只做普通 render sync，让 playbackRate / timeScale watcher 更新 BPM / grid 间距；不能使用 `force` 或 `forceRevision` 触发 `playbackSyncRevision`，否则会让暂停 deck 进入播放同步重锚并出现播放头抽动。暂停 deck 的 `currentSec`、播放头和 viewport 锚点必须保持原值。
- 2026-06-26 BeatSync 暂停转播放修法：从轨 / follower 已开启 BeatSync、暂停状态点击播放时，必须先执行 `nativeTransport.beatsync(deck)`，再从 native snapshot 读取吸附后的 `renderCurrentSec/currentSec` 作为 `playStartSecondsOverride` 启动 renderer 播放时钟。不能在吸附后继续读取吸附前的 `resolveDeckRenderCurrentSeconds(deck)`，否则声音或 render clock 会从旧 grid 相位启动，表现为播放后没有吸附到参照 deck 网格线。
- 2026-06-26 快速暂停再播放复测规律：暂停后立刻播放更容易不对齐，暂停后稍等再播放可以对齐。按 deck 串行化 play/pause 的候选已被复测证伪，不能作为体验规则保留；它还会让快速 pause/play 的第二次输入等待上一轮 async 流程完成，可能带来输入延迟。真正需要保留的规则是 detail lane 的 `stablePlaybackToggleRenderHeld` 不能压过 BeatSync 播放重锚：普通按钮切换防闪可以 hold，但 linked / BeatSync 播放开始、`playbackSyncRevision` 变化、明显播放位置跳变或歌曲变化必须能打破 hold，并以吸附后的秒数重锚 stable canvas / preview viewport。
- 本轮用于定位 BeatSync 暂停转播放的 `[HB-BEATSYNC-PLAY]` 临时落盘诊断已在提交前清理。后续若再次出现偶发旧相位开播，只能按当轮复现短期恢复窄日志；除非维护者明确要求保留，否则不能把这类 trace 型日志随最终提交常驻。
- 这些修法不放宽联结验收标准：新 linked frame ready 后仍必须切到同一轮 `linked-playback` presentation，grid、波形和播放头要进入同一套 viewport / time basis。旧 frame 只允许作为替代帧 ready 前的过渡画面，不能长期掩盖 grid 未对齐。

本轮维护者已复测通过的范围：一条 deck 正在播放、另一条 deck 暂停且已开启 BeatSync 时，暂停 deck 点击播放能按参照 deck grid 吸附开播；快速暂停后立刻播放也不再复现旧相位开播。后续如果同场景再次复现，应优先检查 `playStartSecondsOverride`、`stablePlaybackToggleRenderHeld` 是否被新逻辑绕回。

其它联结 / BeatSync 场景仍需按完整手测规则复验：播放中开启联结、暂停状态开启联结、开启 BeatSync、关闭联结 / BeatSync、拖拽松手后继续播放。尤其要观察点击当下两条大波形是否还会同时停住、主轨开启 BeatSync 时波形是否仍会自己右移、开启联结或从轨 BeatSync 后小节线是否被错误重定义成大节线，BeatSync 开启后拖动从轨松手是否仍有最终可见 grid 对不齐，一条播放一条暂停时点击暂停 deck 的 BeatSync 是否只改变 BPM / grid 间距且播放头不动，以及新 linked / BeatSync frame ready 后 grid 是否出现二次跳动或短暂错位。类型检查只能证明代码能编译，不能把肉眼卡顿判定为已修复。

如果开启联结仍然卡顿，下一轮排查优先入口：

- `horizontalBrowseStableCanvasPresentation.ts`：确认 `tickPlayback` 在 revision handoff 中是否仍出现 `applied === false` 后断链；如需日志，只记录 handoff frame、pending frame、offset、presentable、playbackRaf 是否续上。
- `horizontalBrowseModeShellDetailTransactions.ts` 与 `horizontalBrowseWaveformPresentationCoordinator.ts`：确认 `beginSyncTransaction`、detail lane commit、`finishSyncTransaction` 的 revision 顺序没有让一条 deck 先解除 pending、另一条 deck 后追。
- `useHorizontalBrowseFaderControls.ts`：如果仍必卡，优先检查 native `setLeader` / `setSyncEnabled` / `alignToLeader` 连续 IPC 是否每一步都广播 snapshot 并触发 renderer 重算；同时确认关闭链路的 `after-disable-native-sync` 发生在下一次 `session-begin` 之前。不要重新引入 `force:'all'` 或按钮 loading pending 来掩盖。
- `horizontalBrowseDetailPresentationConsumer.ts` 与 `horizontalBrowseDetailPlaybackPositionWatch.ts`：确认 `linked-playback` 提交后没有被 ordinary playback watcher 抢回 viewport，也没有在 `dualTransportSyncActivating` 的 hold 窗口内重复重锚。
- `useHorizontalBrowseRawWaveformCanvas.ts`、`horizontalBrowseDetailLiveCanvas.worker.ts`：确认替代 frame ready 前没有清空 stable presentation、双缓冲可见面或错误暴露半成品 frame；复测日志里不应再出现 `canvas-preserve-display-ready-stable-revision`。如果仍卡，应按需短期恢复同类窄日志，对照 `canvas-reset-display-ready-stable-revision`、`canvas-skip-duplicate-stable-revision-render`、`canvas-stable-replacement-rendered`、`stable-tick-handoff-applied` 和 `stable-handle-rendered` 的时间窗继续定位。
- 本轮 `[HB-LINK-OPEN]` 临时诊断链已在提交前清理。后续如果再次复现，只能按当轮问题短期恢复窄日志；除非维护者明确要求保留，否则不能随最终提交常驻。

### 手测验收规则

大波形验收必须把“真实操作发生了”和“最终画面正确”分开观察。真实操作必须从界面入口完成，最终结论以维护者看到的画面和听到的声音为准；直接调用内部 handler、直接改 store 或只看 presentation state，不能作为体验验收。

每次手测至少覆盖三类观察：

- 真实操作：从界面上点击联结 / BeatSync、播放、拖拽大波形或滚轮缩放，不用内部状态代替用户动作。
- 最终画面：观察最终合成出来的波形、grid、cue、loop、播放头是否在同一套 viewport / time basis 下移动和对齐，而不是只看按钮状态、transport state 或 computed 值。
- 时间连续性：对播放中场景，要覆盖点击当下、点击后短时间滚动、以及后续稳定滚动。任何阶段出现临时错位、追赶、抖回或二次吸附，都按失败处理。

联结 / BeatSync 的 grid 对齐不能只证明“相位算出来相同”。如果实际 overlay 绘制公式是按 `firstBeatMs`、`bpm`、`rangeStartSec`、`rangeDurationSec` 把 beat time 映射到屏幕 x，那么 `barBeatOffset` 或 presentation phase 只能证明样式相位或理论相位，不能单独证明可见线条已经对齐。最终判断必须以屏幕 x 对齐为准。

联结 / BeatSync 的屏幕 x 对齐和 grid 定义必须分开验收：屏幕上可比较的 beat/grid 线要对齐，但每条 deck 的大节线、小节线和普通线仍按自己的歌曲 grid 定义显示。不能用改写 follower `barBeatOffset` 的方式把小节线伪装成大节线。

这里的屏幕 x 对齐使用完整虚拟 grid 判断。虚拟 grid 可以延伸到歌曲 0 秒之前，理解为用户看不见的节拍尺；双轨联结后，这把节拍尺在上下 deck 上必须对齐。实际渲染仍然必须裁剪负时间线，不能把 0 秒前的虚拟 grid 画出来。

手测屏幕 x 对齐口径：

- 同一轮 sync transaction 后，top / bottom 必须进入同一个显示 revision 或等价的同轮视觉提交；不能一条 deck 先提交，另一条 deck 靠后续 watcher、native snapshot、拖拽或 worker ready 追上。
- 在画布中部可见区域内，top / bottom 可比较的虚拟 beat / bar grid 线应有对应线条落在相同 x 位置。边缘刚进入或刚离开画布的线可以剔除，但不能用边缘剔除掩盖中部错位。
- 肉眼能看到上下 grid 线歪、间距不一致、相位错开或滚动时一条先跑一条追，就按失败处理。
- 至少要在有足够可比较 grid 线的位置手测。只有一两条线、只有边缘线、或某一 deck 没有可见 grid 线时，不能判定“已对齐”，应换到更容易观察的位置重新测。
- 派生指标只能作为排查参考。只要最终画面里 top / bottom 线条实际错开，就按渲染失败处理。

播放中点击联结 / BeatSync 的独立验收口径：

- 前置状态必须是至少一条 deck 正在真实播放，且大波形正在可见滚动；不能用“暂停时开 BeatSync / 联结”“先开 BeatSync / 联结再播放”“播放一小段后停止检查静态状态”替代。
- 点击联结 / BeatSync 的入口必须是真实界面按钮，不能先用内部状态把对齐关系切好再观察画面。
- 点击后的第一轮可见画面就必须进入对应关系。联结要进入双轨整体关系；BeatSync 要进入当前 deck 对齐参照 deck 的 grid 关系。允许画面使用受控旧帧过渡，但旧帧不能展示错误的 grid 对齐关系；不能让用户先看到上下 grid 歪着滚动，再靠下一次拖拽、下一次 snapshot 或下一轮 worker frame 修正。
- 如果 BeatSync 会改变当前 deck 的播放速度、播放位置或视觉 time scale，显示层必须在同一轮事务内提交当前 deck 的 viewport、grid time basis、playback clock 和 visual playback scale；参照 deck 不能被 BeatSync 二次 seek。声音状态已经变化而大波形仍按旧 grid / 旧速度滚动，按失败处理。
- 点击 MASTER / leader deck 的 BeatSync 时，如果 native transport 没有移动音频，主轨大波形也不能出现可见右移。点击联结或 follower deck 的 BeatSync 时，即使 follower 某根小节线正好对齐到 leader 大节线，follower 那根线仍必须保持自己的小节线样式。
- 一条 deck 正在播放、另一条 deck 暂停 / 未播放时，点击暂停 deck 的 BeatSync 不是播放中 phase align 场景。它只能让暂停 deck 改 BPM / playbackRate / grid 间距，不能出现暂停 deck 的播放头、波形 viewport 或 stable frame 锚点追向播放 deck。
- 播放中点击联结后，继续观察滚动至少数秒。若出现一轨先跑、另一轨追上、慢半拍、抖回、两次缩放或两次位置变化，按失败处理，即使最终静止画面看起来对齐。

歌曲起点前空白的独立验收口径：

- viewport 可以显示歌曲 0 秒之前的屏幕区域，但这段区域只能是干净空白或明确的非内容背景。不得出现 grid、节拍刻度、小节线、cue、loop、假波形或任何会被用户理解为歌曲内容的元素。
- 计算层可以把 0 秒之前理解为不可见节拍尺，用于保持联结对齐；但最终画面里 0 秒之前不能画出任何 grid、节拍刻度、小节线、cue、loop 或假波形。
- 这个规则覆盖首次载入、换歌、缓存恢复、空态、worker 首帧返回、seek 到开头附近、播放头居中、点击联结 / BeatSync、联结后拖拽、联结后缩放和 BeatSync 后单轨拖拽松手吸附。不能只在播放稳定后满足。
- 禁止为了让双轨 grid 看起来对齐而在 0 秒之前反推、复制、补画或密集铺设 grid 线。起点前没有音乐时间线内容，不能把它当成对齐缓冲区。

## 总体体验目标

用户进入双轨模式或编辑模式后，应该看到一套稳定、清楚、可用于 DJ 判断的大波形。

第一优先级：

- 声音播放流畅度永远高于波形显示流畅度；任何时候都不能为了让波形更顺牺牲声音稳定。
- 滚动必须流畅。
- 波形必须清楚。
- 操作和播放过程中不能有可感知卡顿。
- 任何时候都不能出现抽搐、闪烁、热气感、抖动、频闪或形变。
- 网格线、cue 点、loop、播放头和波形必须始终作为一个整体滚动，不能有分离、漂移或速度不一致。
- 滚轮调节波形放大缩小时必须流畅，不能有卡顿、跳变或精度切换断裂感。
- 已经有波形的歌曲，用户看到的应是分析完成后的稳定波形，不应在播放、拖拽、seek 或缩放时表现得像还在现场计算波形。
- 开启联结或 BeatSync 时，两条 deck 的网格线必须严格对齐，播放和滚动过程中不能出现临时错位、追赶或漂移；但只有联结允许把双轨 viewport 作为整体拖拽或缩放。
- 用户当前关注的位置必须稳定，缩放、窗口变化、模式切换或布局变化不能把视野突然带飞。
- 播放头必须始终清楚可见，不能被波形、grid、cue 点或 loop 区间遮住。

这个大波形要让用户自然感到：

- 声音始终稳定，波形只是帮助判断，不能反过来影响听感。
- 鼓点和节奏起点清楚。
- 波形形状可信，能帮助判断歌曲结构。
- 播放、拖拽、seek 时画面稳定。
- 时间轴、播放头、grid、cue、loop 位置可靠。
- 暗色和亮色主题下都不费眼。
- 不需要用户安装或打开 Rekordbox，也能获得接近 Rekordbox RGB 大波形的观感。

## 视觉验收

用户看到的双轨和编辑模式大波形必须满足：

- 整体风格接近 Rekordbox RGB 大波形，而不是普通小波形或三段频谱小条。
- 同一首歌、同一位置下，FRKB 的大波形轮廓应和 Rekordbox 的大体形状接近。
- 鼓点、kick、snare、短 loop 的起点要明显，用户能一眼看出节奏落点。
- 波形可以适度平滑，但不能把鼓点起点抹糊、提前或拖尾。
- 颜色不要求和 Rekordbox 完全一样，但高低频和能量变化要有类似的视觉语义。
- 波形内部要有厚度、密度和能量感，不能只剩一条外轮廓。
- 播放头应有最高的定位可读性，用户任何时候都能一眼看出当前播放位置。
- cue 点和 loop 区间要清楚可辨，密集时也不能被波形淹没或抢乱视觉层级。
- 高 DPI 屏幕或 Windows 缩放比例变化下，波形、网格线、cue 点、loop 和播放头仍应清晰，不应发虚、粗细跳变或边缘糊掉。
- 双轨和编辑模式的大波形视觉语言应一致，包括颜色、粗细、grid 层级、cue 表达和播放头优先级，不能像两个产品。
- 亮色主题下不能刺眼，波形区域不能像直接画在白底上一样发飘。
- 暗色主题下不能糊成一团，grid、cue、loop、播放头和波形主体都要分得清。

## 双轨模式体验验收

用户在双轨模式中应该获得这些体验：

- 播放声音必须稳定连续，波形加载、滚动、拖拽、seek、缩放或 sync 变化都不能造成声音卡顿、爆音、断续、变速或瞬间静音。
- 加载一首已有波形数据的歌曲后，不需要先播放或先 seek，就能看到可用的大波形。
- 加载或换歌的第一眼必须是可信画面；如果播放头居中导致左侧露出歌曲起点前的区域，那一段只能是干净空白，不能先出现一堆 grid、节拍刻度或虚拟小节线再被修正。
- 已有波形数据的歌曲进入双轨模式后，波形应作为现成结果直接可用；用户不应感觉每次播放、拖拽或跳转都在重新计算波形。
- 点击概览跳转后，大波形应快速出现在目标位置，不应明显卡住等待。
- 拖拽大波形时，画面必须跟手、流畅，不应卡顿、不可解释的空白、断层、卡死或显示明显旧位置残留。
- 远距离 seek 时，如果新位置波形需要短暂准备，可以出现受控加载态或空态再淡入；但空态中仍应保留时间轴、中心线、播放头或等价结构信息，切换过程不能抽搐、闪烁、跳帧或让用户看到旧位置和新位置来回乱跳。
- 双轨模式 A / D 这类固定 4 beats 近距离热键跳转，以及编辑模式 A / D 与前进 / 后退 beats 按钮这类跟随当前 beat 步长的近距离跳转，应复用当前稳定波形画面；不能触发不可解释的空白、空态、淡入或像重新计算波形一样闪一下。
- 向前回拉时，前方波形不应突然缺失导致滚动停住。
- 播放过程中，大波形应稳定、流畅、清楚地滚动，不应持续抖动、闪烁、热气感变形、卡顿或忽大忽小。
- 播放头位置应可信，用户不应看到波形先飘一段再跳回正确位置。
- grid、cue、loop、播放头和波形应始终一起滚动，不应出现覆盖层和波形分离、漂移、速度不一致或先后错位更新。
- 使用滚轮放大或缩小波形时，缩放过程应连续、顺滑，不应卡顿、闪一下、突然跳级或让波形和覆盖层错开。
- 使用滚轮缩放时，用户鼠标附近或当前关注的鼓点位置应稳定，缩放后不能把关注点带飞。
- 双轨上下两条波形的视觉尺度应稳定，不能因为临时速度、同步或播放状态变化突然膨胀或收缩。
- 开启 BeatSync 后，上下 deck 的网格线应严格对齐；滚动中不能出现某一侧网格线临时跑偏、慢半拍、追上来或抖回去。
- 开启联结或 BeatSync 后，网格线对齐不能改变歌曲自己的线条定义；从轨小节线对齐到主轨大节线时，从轨小节线仍应保持小节线，不能被显示成大节线。
- 开启联结的瞬间，上下 deck 的网格线、波形和播放头关系应一次性进入稳定对齐状态；用户不应看到某一轨先抽动、先缩放、先偏移，然后再贴合到另一轨。
- 点击开启联结或 BeatSync 后，用户不应该还需要拖拽一下大波形才能看到网格线真正对齐；“点一下是歪的，拖一下才齐”视为联结或 BeatSync 对齐事务失败。
- 播放中点击开启联结是独立验收场景，不能用“先联结再播放”代替。前置状态是至少一条 deck 正在播放，用户点击联结的那一帧起，上下 deck 的可见 grid 线、节拍间距、播放头关系和滚动画面必须立即进入同一套联结对齐关系；不能先看到上下 grid 歪着滚动，不能一条 deck 先按旧速度继续跑、另一条 deck 等 native snapshot 或下一次拖拽后才追齐。
- 播放中点击开启 BeatSync 时，如果为了对齐需要调整当前 deck 的播放位置或速度，显示层也必须同一轮提交当前 deck 的 viewport、grid time basis、playback clock 和 visual playback scale。用户不应看到声音已经进入 BeatSync 状态，但大波形 grid 还保持 BeatSync 前的错位画面。
- 播放中点击 MASTER / leader deck 的 BeatSync 时，如果声音位置没有变化，主轨大波形也不应右移；任何“声音没动但波形右移”的结果都视为视听错位。
- 一条 deck 正在播放、另一条 deck 暂停时，点击暂停 deck 的 BeatSync 后，暂停 deck 只允许 BPM / grid 间距发生变化；暂停播放位置、播放头、波形 viewport 和当前可见 anchor 都不应移动，不能出现“暂停轨追播放轨”的动作。
- 从轨 / follower 已开启 BeatSync 且暂停时，点击播放后必须从吸附到参照 deck grid 的位置开始播放；不能偶发停在旧 grid 相位，也不能靠下一帧、下一次 snapshot 或用户再拖拽才对齐。
- 如果某条 deck 在歌曲开头附近，播放头居中可能露出 0 秒之前的空白区域；这段空白必须保持干净，不能出现密集网格线、节拍刻度、虚拟小节线或任何让用户误以为歌曲起点前还有内容的视觉元素。这个规则从载入第一帧开始生效，不能等用户拖拽、播放或下一帧刷新后才恢复正确。
- 开启联结后，拖拽任意一条 deck 的大波形，都应等同于平移双轨整体；上下两条波形、grid、cue、loop 和播放头关系必须一起移动，另一条 deck 不能停住、滞后、等松手才对齐。
- 开启 BeatSync 后，拖拽任意一条 deck 的大波形，只能移动当前被拖动的 deck；另一条 deck 不能跟着整体平移。松手后，当前 deck 必须以松手位置为 anchor 就近吸附到与另一条 deck grid 对齐的位置。
- 开启 BeatSync 后，如果被拖动 deck 原本是 leader，另一条未拖动 deck 应作为吸附参照，最终只移动被拖动 deck；不能反向移动未拖动 deck。
- 开启 BeatSync 后，如果只有从轨开着 BeatSync、主轨 / MASTER 没开 BeatSync，用户拖拽主轨时，从轨应立即退出 BeatSync follower 状态并接管 MASTER，且继续正常播放；主轨拖拽松手只应把主轨对齐到新的 MASTER 参照，不能让从轨继续执行 BeatSync。
- BeatSync 按钮只显示开 / 关。`tempo-only` 等中间锁定状态不能通过闪烁按钮表达，用户不应看到 BeatSync 按钮闪烁。
- 开启联结后，滚轮放大或缩小任意一条 deck，都应等同于缩放双轨整体视图；两条 deck 的可视节拍尺度、当前关注位置、节拍和相位关系必须一起变化，不能缩放后各跑各的，或出现某一轨声音继续但画面停住。
- 开启联结后，拖拽松手或缩放结束不应重新按某个 deck 做二次吸附；应保留操作前已经建立好的相对对齐关系，把整体平移或整体缩放后的结果稳定落到两条 deck 上。
- 开启 BeatSync 后，拖拽松手只允许当前 deck 做一次就近吸附；不允许另一条 deck 被二次吸附，不允许先落到松手位置再明显二次跳动到其它位置。
- 开启 BeatSync 后，拖拽松手的最终可见 grid 必须与吸附后的播放点一致；不能出现 transport 相位日志为 0，但大波形或 overlay 因旧 stable frame、旧 viewport、旧 presentation offset 继续显示出一到数个像素偏差。
- 开启 BeatSync 后拖动从轨松手，最终可见画面必须以 native 吸附后的从轨 anchor 为准；不能出现声音和 transport 已吸附，但从轨大波形、overlay 或播放头仍按松手前旧 viewport 显示，导致用户肉眼看到上下 grid 仍没对齐。
- 开启联结或 BeatSync 后，拖拽松手的第一帧到稳定帧之间，grid 线的视觉定义必须保持稳定；大节线、小节线、普通线不能在同一屏幕位置来回变换，也不能出现某一轨在 release pending 帧里临时推导出新的 barBeatOffset、随后稳定帧又改回来的闪烁。
- 边播放边拖拽、边播放边 seek、边播放边缩放、边播放边开关联结或 BeatSync，都应保持流畅和对齐，不应因为组合操作出现卡顿、闪烁或临时错位。
- 如果声音流畅和波形完整显示发生冲突，应优先保证声音流畅；用户可以接受波形短暂淡入或受控空态，不能接受声音被波形拖卡。

## 编辑模式体验验收

用户在编辑模式中应该获得这些体验：

- 编辑模式播放时，声音也必须稳定连续；放大、缩小、seek 或调整标记不能让声音卡顿、爆音、断续或变速。
- 放大查看时，波形仍有足够细节，不能像低清预览被强行拉大。
- 已有波形数据的歌曲进入编辑模式后，用户应能直接查看和编辑波形细节，不应因为放大、缩小、seek 或播放而看到像现场计算造成的等待、闪烁或重绘抖动。
- full layout 下中心线、上下形态、播放头和 grid 层级清楚。
- 调整 BPM、beat grid、cue 或 loop 时，用户应看到对应标记准确变化，不应看到整条波形无理由闪烁或重绘跳动。
- 播放、暂停、seek、切换位置时，波形、网格线、cue 点、loop 和播放头不能明显错位，滚动速度也不能不一致。
- 滚轮放大缩小时，用户应看到波形细节自然变多或变少，不能卡顿、跳变、闪烁，也不能丢失当前编辑位置。
- 滚轮放大缩小时，当前编辑的鼓点、grid 或 cue 点应留在用户预期位置附近，不能突然跑出视野。
- 编辑时用户要能依赖波形判断节拍、起点和结构，而不是只能靠耳朵猜。
- 编辑模式的大波形不能因为追求双轨模式的轻量显示，而牺牲深度 zoom 下的清晰度。

## 布局变化体验验收

以下变化不能破坏大波形体验：

- 调整窗口大小。
- Windows 显示缩放比例为 125%、150% 或高 DPI 屏幕。
- 双轨模式和编辑模式互相切换。
- 展开或收起侧栏、工具栏、面板。
- 切换亮色和暗色主题。

用户不应看到：

- 波形比例突然变化。
- grid、cue 点、loop 或播放头和波形错位。
- 当前关注位置被带飞。
- 波形短暂抽搐、闪烁、跳帧或重新铺开。
- 线条发虚、粗细忽变或高 DPI 下明显变糊。

## 空态和加载态体验验收

当歌曲还没有可用波形、正在加载或正在生成波形时，用户不应该看到难以理解的空白区域。

可接受体验：

- 用户能明确知道这里是大波形区域。
- 即使真实波形暂时不可用，时间轴结构、中心线或等价提示仍然存在。
- 播放头、cue、loop、grid 等可靠信息仍能显示和移动。
- 显示内容只出现在歌曲真实时长范围内；即使用户拖到 0 秒之前，或载入首帧暂时还没有真实波形，也不能看起来像歌曲起点之前还有波形、网格线、节拍刻度或小节线。
- 从空态或加载态过渡到真实波形时，可以淡入，但不能突然跳、闪、抖，也不能让覆盖层位置变化。

不可接受体验：

- 一片空白，让用户以为界面坏了。
- 播放后占位突然消失，区域变空。
- 显示和真实大波形风格完全不一致的假占位。
- 歌曲起点之前或结束之后出现让用户误判的波形、网格线、刻度、小节线或内容；载入瞬间先出现、随后消失也算不通过。
- 真实波形出现时突然把播放头、grid、cue 点或 loop 带偏。

## 操作手感验收

以下操作必须让用户感到稳定、直接、可预期：

- 点击概览 seek。
- 鼠标拖拽大波形。
- 触屏横向 scrub 大波形。
- 播放中观察滚动。
- 暂停后精确定位。
- 双轨模式上下 deck 切换关注对象。
- 分别开启联结和 BeatSync，观察上下 deck 的网格线对齐。
- 编辑模式放大、缩小和调整节拍位置。
- 滚轮连续放大缩小波形精度。
- 调整窗口大小、切换模式、展开或收起侧栏。
- 边播放边拖拽、seek、缩放或开关联结 / BeatSync。

不可接受体验：

- 为了等待波形显示、刷新或对齐，声音出现卡顿、爆音、断续、变速或瞬间静音。
- 操作后明显延迟才有反馈。
- 载入、换歌或恢复缓存的第一帧就在歌曲起点前铺出密集 grid、节拍刻度或虚拟小节线。
- 波形突然变高、变矮、错位或重置。
- 拖动过程中画面断裂、出现不可解释的空白或卡住。
- 拖动过程中出现卡顿、不可解释的空白、抽搐、闪烁或旧位置残留。
- 远距离 seek 后出现抽搐、闪烁、来回跳帧或旧位置和新位置混在一起。
- 播放或拖动时出现热气感、持续抖动或卡顿。
- 网格线、cue 点、loop、播放头和波形分开移动，或滚动速度看起来不一致。
- 滚轮缩放波形时出现卡顿、跳级、闪烁、当前位置丢失或覆盖层错位。
- 已有波形数据的歌曲在播放、拖拽、seek 或缩放时表现得像还在现场计算波形。
- 联结或 BeatSync 开启时，上下 deck 网格线静态对齐但滚动中临时错位、漂移、追赶或抖回。
- 联结或 BeatSync 开启时，上下 deck 的 beat/grid 线对齐了，但 follower 的小节线、大节线或普通线定义被参照 deck 改写。
- 点击联结 / BeatSync 后网格线仍然歪，必须再拖拽一下波形才真正对齐。
- 歌曲起点前的空白区域出现密集网格线、节拍刻度、虚拟小节线或任何假内容。
- 联结开启时，拖拽或缩放任意一条 deck 后，两条 deck 没有像一个整体一样平移或缩放。
- BeatSync 开启时，拖拽任意一条 deck 后，另一条 deck 被错误地整体平移或二次 seek；或者松手后当前 deck 没有按松手位置就近吸附到参照 deck grid。
- BeatSync 拖拽松手后，当前 deck 已经按 native transport 吸附，但最终屏幕上仍有一到数个像素的 grid 错位，或旧波形帧停在松手前位置。
- 点击 MASTER / leader deck 的 BeatSync 后，声音没有对应移动但主轨波形可见右移。
- 联结开启时，拖拽或缩放过程中出现另一条 deck 停住、滞后、松手前不跟随、松手后错位，或某一轨声音继续但画面停住。
- 缩放、窗口变化或模式切换后，用户当前关注的鼓点、grid 或 cue 点被突然带飞。
- 连续快速操作时出现卡顿、抽搐、闪烁、错位或新旧画面混乱。
- 高 DPI 或系统缩放下线条发虚、粗细跳变、cue 点糊掉。
- 播放头和听到的声音不同步。
- 用户刚定位的位置被后续刷新带跑。

## 声音优先级验收

声音是最高优先级体验。

用户播放歌曲时，无论波形处于加载、滚动、缩放、seek、拖拽、sync、模式切换还是主题切换中，都必须保证：

- 声音连续。
- 声音不爆音。
- 声音不忽快忽慢。
- 声音不因为波形显示暂停、等待或重算。
- 声音和播放头保持可信同步。

可接受体验：

- 远距离 seek 后，波形进入受控加载态或空态再淡入。
- 波形暂时不完整，但声音稳定播放。

不可接受体验：

- 为了让波形先准备好而延迟声音。
- 波形卡住时声音也跟着卡住。
- 缩放或拖拽波形导致声音爆音、断续、变速或瞬间静音。

## Rekordbox 参考验收

Rekordbox 是视觉参考，不是运行依赖。

用户不需要安装、打开或分析 Rekordbox，也应该在 FRKB 里看到接近 Rekordbox RGB 大波形的体验。

验收时可以用 Rekordbox 同歌同位置作为参考图，但判断重点是：

- 鼓点起点是否同样清楚。
- 简单节奏的轮廓是否接近。
- 高能量段和低能量段是否容易分辨。
- 同类声音在 FRKB 中是否呈现出相近的形状关系。
- 是否存在 FRKB 明显比 Rekordbox 更糊、更飘或更难判断节奏的情况。

## 歌曲覆盖验收

验收不能只看短 loop 或简单歌曲，还要覆盖：

- 普通 5-8 分钟歌曲。
- 较长歌曲。
- 动态变化明显的歌曲。
- 鼓点简单清楚的歌曲。
- cue 点和 loop 较多的歌曲。
- 高能量段和低能量段差异明显的歌曲。

这些歌曲上都应保持大波形清楚、滚动流畅、覆盖层同步、播放头可读。

## 主题验收

必须分别检查亮色和暗色主题。

亮色主题下：

- 波形区域不能刺眼。
- RGB 波形不能发白、发灰到失去层次。
- grid、cue、loop、播放头不能被浅色背景吃掉。

暗色主题下：

- 波形不能糊成一片高饱和色块。
- grid、cue、loop、播放头不能抢过波形主体。
- 长时间观察不能有明显疲劳感。

## 推荐用户验收流程

这套流程以后按维护者手测执行。只出现短暂载入、播放一小段然后停止，或者只测“先联结后播放”而没有测“播放中点联结”，不算完成验收。

1. 选择简单鼓点、短 loop、普通 5-8 分钟歌曲各几首。
2. 在双轨模式载入歌曲，确认不播放也能看到可用大波形；载入第一眼就要检查歌曲起点前是否干净，不能出现密集 grid、节拍刻度、虚拟小节线或假内容。
3. 点击概览跳转多个位置，观察大波形是否快速、稳定出现。
4. 做远距离 seek，确认允许受控加载态或空态加淡入，但不能抽搐、闪烁、跳帧或新旧位置混乱。
5. 播放中在双轨模式连续按 A / D 做固定 4 beats 近距离热键跳转；在编辑模式切换 beat 步长后分别按 A / D 和点击前进 / 后退 beats 按钮，确认两者跳转距离一致，且波形和覆盖层直接稳定跳到目标位置，不出现不可解释的空白、不空态、不淡入、不闪一下。
6. 拖拽大波形，特别是向前回拉，观察是否跟手、流畅、不断层、不出现不可解释的空白、不卡顿。
7. 重复播放、拖拽、seek 和缩放，确认已有波形数据的歌曲不会表现得像还在现场计算波形。
8. 播放至少 30 秒，观察是否稳定滚动，无持续抖动、抽搐、闪烁、热气感或缩放变化。
9. 检查 grid、cue、loop、播放头和波形是否始终作为一个整体滚动，不能分离、漂移或速度不一致。
10. 在未播放或暂停状态下分别开启联结和 BeatSync，观察上下 deck 网格线是否严格对齐；点击开启后应立即对齐，不能需要再拖拽一下波形才贴齐。
11. 在至少一条 deck 正在播放时分别点击开启联结和 BeatSync，观察点击当下和随后滚动中的上下 deck 网格线是否立即对齐。不能出现点击后先歪着滚、某一轨慢半拍、追上来、抖回去，或声音已经进入联结 / BeatSync 但大波形 grid 还停留在开启前错位关系；点击 MASTER / leader deck 的 BeatSync 时，还要确认声音没有移动时主轨波形也没有右移。
12. 一条 deck 正在播放、另一条 deck 暂停时，点击暂停 deck 的 BeatSync，确认暂停 deck 只改变 BPM / grid 间距，播放头、当前可见 anchor、波形 viewport 和 stable frame 锚点都不移动，也不做追播放 deck 的动作。
13. 从轨 / follower 已开启 BeatSync 且暂停时，点击播放，确认播放开始位置已经吸附到参照 deck grid；不能偶发旧相位开播，不能播放后再追齐。
14. 在其中一条 deck 接近歌曲开头时开启联结或 BeatSync，确认 0 秒之前的空白区域从开启瞬间开始就没有网格线、节拍刻度、虚拟小节线或假内容，不能先歪或先铺满再等拖拽后修正。
15. 开启联结后，拖拽任意一条 deck 的大波形，确认两条 deck 像一个整体一样平移，另一条 deck 不停住、不滞后、不等松手才对齐。
16. 开启 BeatSync 后，拖拽任意一条 deck 的大波形，确认拖动期间只有当前 deck 跟手移动；松手后当前 deck 按松手位置就近吸附到与另一条 deck grid 对齐，另一条 deck 不被拖动、不被二次 seek。
17. 联结开启后和 BeatSync 拖拽松手后，继续看当前 deck 的最终稳定画面，确认网格线不是只在 transport phase 上对齐，而是在屏幕 x 坐标上也没有一到数个像素的旧帧偏移；同时找一处“leader 大节线对齐 follower 小节线”的位置，确认 follower 小节线没有变成大节线。如果出现轻微错位，应优先检查 release viewport、stable frame reuse 和 presentation offset，而不是先改 BPM 或 native align。
18. 开启联结后，使用滚轮连续放大缩小任意一条 deck，确认两条 deck 像一个整体一样缩放，缩放后相对节拍和网格关系不跑散。
19. 使用滚轮连续放大缩小波形，确认缩放流畅、不卡顿、不跳级、不闪烁，覆盖层和波形仍然对齐。
20. 确认滚轮缩放时，鼠标附近或当前关注的鼓点、grid、cue 点不会被带飞。
21. 边播放边拖拽、seek、缩放或开关联结 / BeatSync，确认组合操作下也不闪、不抖、不错位。
22. 边播放边执行波形拖拽、seek、缩放、联结、BeatSync、模式切换，确认声音不因波形操作卡顿、爆音、断续、变速或瞬间静音。
23. 调整窗口大小，展开或收起侧栏，确认波形比例和覆盖层不乱跳。
24. 切到编辑模式，放大查看鼓点和 grid，确认细节足够用于编辑判断。
25. 调整 BPM、beat grid、cue 或 loop，确认标记变化清楚，波形不乱跳。
26. 分别切换亮色和暗色主题，确认波形区域、grid、cue、loop、播放头都清楚。
27. 在高 DPI 或 Windows 125%/150% 缩放下检查线条、cue 点、loop 和播放头是否清晰。
28. 对照 Rekordbox 同歌同位置参考图或 contact sheet，确认大体形状和鼓点起点接近。

## 一票否决项

出现以下任一用户可感知问题，本次大波形验收不通过：

- 为了波形显示、滚动、缩放、seek、拖拽、联结或 BeatSync，声音出现卡顿、爆音、断续、变速或瞬间静音。
- 鼓点起点明显糊、提前或拖尾。
- 拖拽大波形时出现不可解释的空白、断层、卡顿或旧位置残留。
- 远距离 seek 后出现抽搐、闪烁、来回跳帧或新旧位置混乱。
- 双轨模式 A / D 固定 4 beats 近距离热键跳转，或编辑模式 A / D 与前进 / 后退 beats 按钮触发不可解释的空白、空态、淡入、明显重绘闪烁，或跳转距离不一致。
- 任何时候出现明显抽搐或闪烁。
- 播放过程中持续抖动、频闪、热气感、卡顿或缩放变化。
- 网格线、cue 点、loop、播放头和波形出现分离、漂移或滚动速度不一致。
- 滚轮放大缩小时出现明显卡顿、跳变、闪烁、当前位置丢失或覆盖层错位。
- 已有波形数据的歌曲在播放、拖拽、seek 或缩放时仍表现出现场计算造成的等待、闪烁、重绘抖动或卡顿。
- 联结或 BeatSync 开启时上下 deck 网格线没有严格对齐，或滚动中出现临时错位、漂移、追赶、抖回。
- 点击开启联结或 BeatSync 时出现明显的先抽动再贴合、两次位置变化、临时错位或追赶感。
- 点击 MASTER / leader deck 的 BeatSync 后，声音没有对应移动但主轨波形右移，导致声音和波形错位。
- 一条 deck 播放、另一条 deck 暂停时，点击暂停 deck 的 BeatSync 导致暂停 deck 播放头 / 波形 viewport / stable frame 锚点移动，或者暂停 deck 做出追播放 deck 的动作。
- 从轨 / follower 已开启 BeatSync 且暂停时，点击播放后没有从吸附到参照 deck grid 的位置开始播放，或播放后才追齐。
- 点击联结或 follower deck 的 BeatSync 后，follower 原本的小节线或普通线被错误重定义成大节线。
- 点击开启联结或 BeatSync 后网格线仍歪，只有拖拽一下大波形才对齐。
- 播放中点击开启联结或 BeatSync 后，上下 deck 网格线没有立即对齐，或播放滚动中出现某一轨先按旧 grid / 旧速度继续跑、另一轨随后追齐。
- 载入、换歌、缓存恢复、worker 首帧返回或点击联结 / BeatSync 时，歌曲起点前出现网格线、节拍刻度、虚拟小节线、假波形或其它会让用户误判起点前仍有内容的视觉元素。
- 联结开启时，拖拽或缩放任意一条 deck 没有整体作用到双轨，导致另一条 deck 停住、滞后、松手前不跟随、缩放后跑散，或声音继续但画面停住。
- BeatSync 开启时，拖拽任意一条 deck 导致另一条 deck 跟随平移，或松手后不是当前 deck 按松手位置就近吸附，而是另一条 deck 被二次移动、二次吸附。
- 只有从轨开着 BeatSync、主轨 / MASTER 没开 BeatSync 时，拖拽主轨后从轨仍保持 BeatSync follower 状态、继续执行 BeatSync，或没有接管 MASTER 继续正常播放。
- BeatSync 按钮出现闪烁，或 UI 试图用闪烁表达 `tempo-only` / 半锁定 / 第三种 BeatSync 状态。
- BeatSync 拖拽松手后，native transport 或日志显示已经对齐，但最终屏幕上的大波形、grid 或 overlay 仍因旧 stable frame、旧 viewport 或旧 presentation offset 出现一到数个像素偏差。
- 缩放、窗口变化或模式切换后当前关注位置被带飞。
- 连续快速操作时出现卡顿、抽搐、闪烁、错位或新旧画面混乱。
- 播放头被波形、grid、cue 点或 loop 遮挡，导致用户无法一眼看出当前位置。
- 高 DPI 或系统缩放下波形、网格线、cue 点、loop、播放头明显发虚或粗细跳变。
- 空态或加载态切到真实波形时出现跳动、闪烁、覆盖层位置变化。
- 播放头、grid、cue、loop 和声音或时间轴明显不同步。
- 亮色或暗色主题任一主题下不可读。
- 编辑模式深度 zoom 下细节明显不足，无法用于节拍编辑判断。
- 需要用户依赖 Rekordbox 才能获得可用大波形体验。
