# 双轨和编辑模式大波形体验与架构标准

维护状态：长期维护

更新时间：2026-06-26

## 文档定位

这个文档是双轨模式和编辑模式大波形的长期维护标准。它约束用户体验、显示 owner、渲染链路、缓存链路、overlay、sync、seek、drag、zoom、grid 编辑和主题适配。

本文不是一次性问题复盘，也不是某次修复的提交说明。历史排查只保留已经沉淀成规则的经验结论，禁止把复现过程、临时日志和候选补丁按时间顺序继续堆在正文里。

验收只看用户能看到、能操作、能感受到的结果。只要维护者肉眼看到联结、BeatSync、拖拽、缩放、播放、起点前空白或 grid 对齐存在问题，即使内部状态看起来正确，本项也按不通过处理。

维护本文时只写三类内容：

- 长期不变量：声音优先级、波形清晰度、覆盖层同步、sync 对齐、播放头可信度、主题和 DPI 可读性。
- 当前架构边界：当前 checkout 的真实模块职责、owner 入口、presentation handoff 和禁止回退项。
- 验收规则：用户视角的通过 / 不通过标准、推荐手测流程和一票否决项。

## 核心原则

- 声音优先级最高。任何波形加载、滚动、缩放、seek、拖拽、联结或 BeatSync 行为都不能让声音卡顿、爆音、断续、变速或瞬间静音。
- 波形、grid、cue、loop、播放头必须使用同一套 viewport、time basis、zoom 和 playback clock 派生，不能各算一套屏幕坐标。
- 同一时刻只能有一个显示 owner 提交大波形的可见位置。其它层可以提供数据、快照或候选结果，但不能抢着改最终画面。
- 联结和 BeatSync 都是可见 grid / playback 关系对齐，不是重写歌曲自身 grid 定义。`firstBeatMs`、`barBeatOffset`、大节线 / 小节线层级和 phrase 语义必须属于各自歌曲。
- 已有波形数据的歌曲在播放、拖拽、seek、缩放或 sync 中不应表现得像还在现场计算波形。
- 手测是唯一验收通过口径。内部日志、computed 值、native phase 和工具输出只能辅助定位，不能把维护者看到的错位、闪烁、抽动、卡顿改判为通过。

## 术语边界

- 联结：双轨联结按钮开启后的双 deck 视图联动。联结开启后，拖拽或缩放任意一条 deck 都等同于操作双轨整体，两条 deck 的 viewport、grid、cue、loop 和播放头关系必须同轮更新。
- BeatSync：单条 deck 的 BeatSync 按钮。BeatSync 只表示当前 deck 与另一条 deck 做 tempo / beat-grid 对齐，不表示双轨 viewport 联动，也不表示拖拽时两条 deck 要一起平移或缩放。
- BeatSync 按钮只允许表达开 / 关两种状态。`tempo-only`、phase 未完全锁住或其它中间状态不能让按钮闪烁，也不能让用户误以为还有第三种 BeatSync 模式。
- 联结和 BeatSync 的 grid 对齐只表示可见 beat/grid 位置和播放相位进入对齐关系。如果主轨的大节线对齐的是从轨的小节线，从轨那根线仍必须保持小节线。
- 本文凡写“联结后整体拖拽 / 整体缩放”，只适用于联结按钮语义，不适用于 BeatSync。

## 架构边界

### Surface

FRKB 里和波形相关的 surface 至少包括：

- 歌曲列表波形预览。
- 浏览器模式主播放器波形。
- 双轨模式 overview 波形。
- 编辑模式 overview 波形。
- 双轨模式 detail 大波形。
- 编辑模式 detail 大波形。

这些 surface 可以复用缓存、颜色、渲染工具或视觉语言，但不能默认共用运行时状态机。跨 surface 复用必须先说明复用的是数据、样式、渲染函数还是交互状态，禁止把一个 surface 的播放、seek、拖拽、缩放状态直接打进另一个 surface。

### 事实源分层

- 音频事实源：native transport 负责声音播放、暂停、seek、同步倍率、leader / follower、playhead 加载状态。
- 歌曲时间线事实源：song grid、BPM、first beat、bar beat offset、time basis、cue、loop 决定音乐事件在歌曲时间上的位置。
- 波形数据事实源：已分析波形缓存、统一 display waveform、compact visual、detail waveform 只描述可绘制的波形形状。
- 视口和 presentation 事实源：当前用户关注哪一段、播放头显示在哪里、可视窗口宽度和缩放比例，由当前显示 owner 决定。
- 覆盖层事实源：grid、cue、loop、播放头从同一个时间线和同一个 viewport 映射得到。
- 主题和 DPI 事实源：主题、颜色、线宽、device pixel ratio 只能影响视觉样式和清晰度，不能改变歌曲时间、播放位置或对齐关系。

grid 可以在计算层作为虚拟时间线向歌曲 0 秒之前延伸，用于保证双轨联结时完整节拍尺对齐；但最终画面必须裁剪到歌曲真实时间线内，不能在起点前画出 grid、刻度、cue、loop 或假波形。

### 当前主链路

当前 detail 大波形主链路是：

`用户操作 intent / transport 校准 / waveform data ready` -> `HorizontalBrowseModeShell.vue` -> `WaveformPresentationCoordinator` -> `detail lane presentationState` -> `HorizontalBrowseRawWaveformDetail.vue` -> `canvas renderer / worker`

核心边界：

- shell 负责跨 deck 事务入口。
- coordinator 负责 owner、revision 和双轨 presentation state。
- detail lane 负责透传和局部交互。
- raw waveform detail 负责消费 state 并发出 pointer / wheel / grid edit intent。
- canvas / worker 负责按 state 绘制，不能决定跨 deck presentation owner。

### 关键模块职责

- `HorizontalBrowseModeShell.vue`：组合双轨 surface，接入播放、seek、drag、zoom、sync 和 stable frame 事务。
- `horizontalBrowseWaveformPresentationCoordinator.ts`：维护 top / bottom presentation state，统一提交 owner、revision、source deck、affected decks、viewport、zoom、time scale、surface mode、linked 和 visual pending。
- `horizontalBrowseModeShellDetailTransactions.ts`：承接 sync visual transaction、stable frame 准备和 detail lane 视觉事务桥接。
- `HorizontalBrowseDeckDetailLane.vue`：透传 stable frame、linked visual commit 和 presentation state，不拥有跨 deck presentation 决策权。
- `HorizontalBrowseRawWaveformDetail.vue`：消费 presentation state，维护 detail 本地网格和交互入口；不能绕过 coordinator 决定双轨最终 viewport。
- `horizontalBrowseDetailPresentationConsumer.ts`：按 coordinator revision 应用 seek、zoom、viewport 和 anchor。
- `horizontalBrowseDetailPlaybackPositionWatch.ts`：处理 playback / currentSeconds 校准、稳定播放帧复用和 seek handoff；presentation transaction 或 visual pending 存在时不能抢回 viewport owner。
- `useHorizontalBrowseRawWaveformCanvas.ts`：管理 canvas renderer、frame cache、可见性、worker bridge 和 viewport 计算；不能成为跨 deck presentation owner。
- `useHorizontalBrowseDeckPlaybackController.ts`：负责 transport、scrub、seek intent、BeatSync 播放前吸附和播放状态切换；不能直接决定最终 detail viewport。
- `useHorizontalBrowseTransportMutations.ts`：负责联结 / BeatSync 开关语义；必须区分播放中 phase align 与暂停 deck tempo-only。

## Owner 规则

### 通用 owner

- 普通播放：稳定 presentation / canvas 播放循环拥有可见移动；`currentSeconds` watcher 只做校准。
- seek：seek intent 是显示 owner；近距离 seek 应复用当前稳定画面，远距离 seek 可以请求新帧或短暂受控空态。
- 拖拽：pointer / scrub 交互是显示 owner；native snapshot 或播放状态回传不能在拖拽中把 viewport 拉回旧位置。
- 缩放：wheel intent 提交 zoom、anchor 和 viewport；联结缩放必须同轮更新两条 deck 的可视时间尺度和关注位置。
- grid / BPM / cue / loop 编辑：编辑交互是时间线事实源；波形本体、overlay、播放头必须使用同一 viewport / time basis 派生。
- worker / cache ready：只能提升某个 revision 的可绘制帧，不能反向改变用户当前关注位置。

### 联结 owner

- 联结开启时，shell / fader 层的一次性视觉事务是显示 owner；上下 deck 的 grid、视觉节拍尺度、viewport anchor 必须同一轮提交。
- 开启联结时，visual transaction 必须先于 native sync / align 的可见状态变化进入 pending。不能等 native 已经改完 tempo、leader、syncLock 或播放位置后，再让 detail lane 事后追赶新关系。
- 联结拖拽和联结缩放都必须以 top / bottom 同一轮 revision 提交，不能恢复为“源 deck 先动，另一条 deck 后追”。
- 联结关闭时必须等待 native sync 关闭完成，再按关闭后的 snapshot 清 linked presentation；不能 fire-and-forget 后允许下一次开启抢跑。

### BeatSync owner

- BeatSync 开启时，BeatSync 对齐事务是显示 owner；它必须让当前 deck 和参照 deck 的 grid 进入对齐关系，但不能把两个 deck 误切成联结拖拽或联结缩放状态。
- 点击 MASTER / leader deck 的 BeatSync 时，如果 native transport 没有移动音频，renderer 也不能移动主轨大波形。
- 点击 follower deck 的 BeatSync 时，leader lane 只能 compute-only，不能被二次 seek、重锚、修改 `previewStartSec`、timeScale 或 stable frame。
- BeatSync 拖拽时，只允许当前被拖动 deck 跟手移动；另一条 deck 是对齐参照，不是被拖拽对象。
- BeatSync 拖拽松手时，只能移动被拖动 deck：以用户松手位置为 anchor，就近吸附到能与参照 deck grid 对齐的位置。
- 如果被拖动 deck 原本是 leader，松手吸附时应以另一条静止 deck 为参照，使被拖动 deck 对齐过去；不能反向移动另一条 deck。
- 如果只有 follower 开着 BeatSync、leader / MASTER deck 没开 BeatSync，而用户拖拽 leader，原 follower 必须退出 BeatSync 并接管 MASTER，继续正常播放。
- BeatSync 松手后最终可见 grid 必须与 native 吸附后的播放点一致；不能出现 transport 相位正确但屏幕上一到数个像素仍错位。

### 暂停 deck 的 BeatSync

- 一条 deck 正在播放、另一条 deck 暂停 / 未播放时，点击暂停 deck 的 BeatSync 只能进入 tempo-only。
- tempo-only 允许暂停 deck 的 BPM / playbackRate / grid 间距变化，但不能修改暂停 deck 的 currentSec、播放头、viewport、stable frame 锚点或当前可见 anchor。
- 该场景不能调用 native `alignToLeader`，也不能开启 BeatSync visual transaction；正确路径是 `setSyncEnabled(deck, true)` 后普通 render sync。
- 从轨 / follower 已开启 BeatSync 且暂停时，点击播放必须先执行 `nativeTransport.beatsync(deck)`，再从 native snapshot 读取吸附后的 `renderCurrentSec/currentSec` 作为 `playStartSecondsOverride` 启动 renderer 播放时钟。
- 吸附后禁止继续读取吸附前的 `resolveDeckRenderCurrentSeconds(deck)` 启动播放时钟，否则声音或 render clock 会从旧 grid 相位启动。
- 快速暂停再播放时，detail lane 的 `stablePlaybackToggleRenderHeld` 不能压过 BeatSync 播放重锚。普通按钮切换防闪可以 hold，但 linked / BeatSync 播放开始、`playbackSyncRevision` 变化、明显播放位置跳变或歌曲变化必须能打破 hold。

## Stable Frame 和渲染规则

- `displayReady` 只表示当前 revision 的新 frame 是否 ready，不能拿来伪装旧 surface 可见性。
- `preserveSurfaceUntilNextReady` 只表示 DOM surface 是否继续可见；`stableRevisionHandoffSurfaceActive` 只表示旧 stable frame 是否可作为短暂滚动 handoff。
- `preserveDisplay` 必须真的保留当前 stable presentation、rendered range 和双缓冲可见面，直到替代帧可展示。禁止只保留 DOM opacity，却把 stable canvas playback、presentation frame 或 revision handoff 清空。
- revision handoff 期间，旧 frame 可以按自身 `viewportRangeStartSec / anchorSec / playbackRate` 继续推进画面，不能把新 linked viewport 直接套到旧 frame 上导致 `presentable=false` 后断掉 playback RAF。
- 旧 stable frame 必须携带生成时的 `playbackActive`。播放态旧帧可以按旧帧时钟 handoff；非播放态旧帧只有在旧 anchor 接近新 pending anchor 时，才能用新的 playback clock 做短暂 static-to-playback handoff。
- 替代帧 pending 期间，如果同一个 stable revision 已经有 worker render 在飞，后续低价值重复 draw 应跳过，避免 token 被连续重置、延长用户看到的停顿窗。
- `visualPending` 只能挡住错误的半成品画面，不能成为可感知冻结的借口。native、worker 或双 lane commit 还没完成时，显示层仍应保持一个语义正确的可展示画面继续滚动，或者使用明确受控的短过渡。
- 继续优化联结开启瞬间轻微卡顿，不应再做 watcher 级小补丁；下一步是更重的架构方向：预构建 linked replacement frame、把 worker stable frame 渲染做成可抢占 / 增量，或让 linked 开启前就持有可直接切换的双 deck 对齐 frame。

## 禁止回退项

这些方向已经证明会造成体验回归或架构混乱，后续不能重新引入：

- 不能用 `displayReady=true` 保旧面。它会让旧 stable frame、pending linked frame、overlay 和 playback watcher 同时认为自己可提交可见状态，曾造成波形缺块、卡顿和视听错位。
- 不能在开启联结后无条件 `syncDeckRenderState({ force: 'all' })`。force 会制造无意义的 render clock 重锚和 playback revision bump。
- 不能把 `dualTransportSyncActivating` 这类按钮 loading 状态直接传给 detail lane 当 `linkedGridVisualPending`。按钮 loading 只属于按钮状态。
- 不能用改写 follower `barBeatOffset` 的方式制造联结 / BeatSync 对齐。屏幕线条可以对齐，线条定义不能被改写。
- 不能把 BeatSync 拖拽松手做成“先提交鼠标松手点，再等 native align 后二次跳”。显示层必须以 native 吸附后的 anchor 作为最终 presentation 事实源。
- 不能把 play/pause 串行队列当作 BeatSync 暂停开播修法。该候选已被复测证伪，还会让快速 pause/play 的第二次输入等待上一轮 async 流程完成，可能带来输入延迟。
- 不能提交常驻 trace 型落盘日志。`[HB-LINK-OPEN]`、`[HB-BEATSYNC-PLAY]` 等临时诊断只能按当轮复现短期恢复；除非维护者明确要求保留，否则提交前必须清理。
- 不能为了波形先稳定而暂停、延迟、重启或拖慢声音。
- 不能只修当前 deck，不检查另一条 deck、overview、overlay、theme、DPI 和编辑模式是否被带坏。

## 已验证结论

- 一条 deck 正在播放、另一条 deck 暂停时，点击暂停 deck 的 BeatSync 只允许 tempo-only；复测通过。
- 从轨 / follower 已开启 BeatSync 且暂停时，点击播放能按参照 deck grid 吸附开播；快速暂停后立刻播放不再复现旧相位开播；复测通过。
- BeatSync 暂停开播的关键实现点是 `playStartSecondsOverride`：使用 `nativeTransport.beatsync(deck)` 后 native snapshot 的 `renderCurrentSec/currentSec` 启动 renderer 播放时钟。
- `stablePlaybackToggleRenderHeld` 必须允许 linked / BeatSync 播放开始、`playbackSyncRevision` 变化、明显播放位置跳变或歌曲变化打破 hold。
- 开启联结瞬间的轻微卡顿已收窄到 stable revision replacement 首帧成本。当前窄修已经避免明显跳动和视听错位；继续优化需要更重的 frame 预构建或 worker 渲染架构调整。

## 修改前检查清单

任何触及双轨或编辑模式大波形的改动，至少先回答这些问题：

- 这次改的是哪个 surface：列表预览、主播放器、双轨 overview、编辑 overview、双轨 detail，还是编辑 detail？
- 这次改的是哪个层：音频 transport、歌曲时间线、波形数据、viewport / presentation、overlay，还是主题 / DPI？
- 当前用户操作的显示 owner 是谁：播放、seek、拖拽、缩放、联结事务、BeatSync 事务、grid 编辑，还是加载态？
- 这次改动是否会让两个 owner 同时提交 viewport？
- 这次改动是否会让波形本体和 grid / cue / loop / 播放头分开更新？
- 这次改动是否会让 native snapshot 或 worker ready 结果覆盖用户当前关注位置？
- 这次改动是否会影响声音连续性、播放头可信度、联结 / BeatSync 对齐、联结整体拖拽、联结整体缩放、BeatSync 单轨拖拽或编辑模式深度 zoom？
- 是否需要短期落盘诊断？如果需要，是否走 `log.txt` 链路、范围足够窄、提交前清理？

## 总体体验目标

用户进入双轨模式或编辑模式后，应该看到一套稳定、清楚、可用于 DJ 判断的大波形：

- 声音稳定，波形只是帮助判断，不能反过来影响听感。
- 鼓点和节奏起点清楚，能帮助判断歌曲结构。
- 播放、拖拽、seek、缩放时画面稳定。
- 时间轴、播放头、grid、cue、loop 位置可靠。
- 暗色和亮色主题下都不费眼。
- 不需要用户安装或打开 Rekordbox，也能获得接近 Rekordbox RGB 大波形的观感。

第一优先级：

- 滚动流畅。
- 波形清楚。
- 操作和播放过程中不能有可感知卡顿。
- 不能出现抽搐、闪烁、热气感、抖动、频闪或形变。
- 网格线、cue 点、loop、播放头和波形必须始终作为一个整体滚动。
- 滚轮调节波形放大缩小时必须流畅，不能有卡顿、跳变或精度切换断裂感。
- 开启联结或 BeatSync 时，两条 deck 的网格线必须严格对齐；但只有联结允许把双轨 viewport 作为整体拖拽或缩放。
- 播放头必须始终清楚可见，不能被波形、grid、cue 点或 loop 区间遮住。

## 验收标准

### 视觉

- 整体风格接近 Rekordbox RGB 大波形，而不是普通小波形或三段频谱小条。
- 同一首歌、同一位置下，FRKB 的大波形轮廓应和 Rekordbox 的大体形状接近。
- 鼓点、kick、snare、短 loop 的起点要明显。
- 波形可以适度平滑，但不能把鼓点起点抹糊、提前或拖尾。
- 波形内部要有厚度、密度和能量感，不能只剩一条外轮廓。
- cue 点和 loop 区间要清楚可辨，密集时也不能抢乱视觉层级。
- 高 DPI 或 Windows 缩放比例变化下，波形、grid、cue、loop 和播放头仍应清晰。
- 双轨和编辑模式的大波形视觉语言应一致。
- 亮色主题下不能刺眼；暗色主题下不能糊成一团。

### 双轨模式

- 加载已有波形数据的歌曲后，不需要先播放或先 seek，就能看到可用大波形。
- 加载或换歌第一眼必须可信；歌曲起点前区域只能是干净空白。
- 点击 overview 跳转后，大波形应快速、稳定出现在目标位置。
- 远距离 seek 可以出现受控加载态或空态再淡入，但不能抽搐、闪烁、跳帧或新旧位置混乱。
- A / D 固定 4 beats 近距离热键跳转应复用当前稳定波形画面，不能空态、淡入或闪一下。
- 拖拽大波形必须跟手、流畅，不应卡顿、断层、不可解释的空白或旧位置残留。
- 播放过程中，大波形应稳定、流畅、清楚地滚动。
- grid、cue、loop、播放头和波形应始终一起滚动，不应分离、漂移、速度不一致或先后错位更新。
- 使用滚轮缩放时，用户鼠标附近或当前关注的鼓点位置应稳定，缩放后不能把关注点带飞。
- 开启联结后，拖拽或缩放任意一条 deck 都应整体作用到双轨。
- 开启 BeatSync 后，拖拽任意一条 deck 只能移动当前 deck；松手后当前 deck 按松手位置就近吸附到参照 deck grid。
- BeatSync 开启后如果只有 follower 开着 BeatSync、MASTER 没开 BeatSync，用户拖拽 MASTER 时 follower 应退出 BeatSync 并接管 MASTER。
- 边播放边拖拽、seek、缩放或开关联结 / BeatSync，都应保持流畅和对齐。

### 播放中联结 / BeatSync

- 播放中点击开启联结是独立验收场景，不能用“先联结再播放”替代。
- 点击联结 / BeatSync 的入口必须是真实界面按钮，不能先用内部状态把对齐关系切好再观察画面。
- 点击后的第一轮可见画面就必须进入对应关系。不能先看到上下 grid 歪着滚动，再靠下一次拖拽、下一次 snapshot 或下一轮 worker frame 修正。
- 如果 BeatSync 改变当前 deck 的播放速度、播放位置或视觉 time scale，显示层也必须同一轮提交当前 deck 的 viewport、grid time basis、playback clock 和 visual playback scale。
- 点击 MASTER / leader deck 的 BeatSync 时，如果 native transport 没有移动音频，主轨大波形也不能右移。
- 一条 deck 正在播放、另一条 deck 暂停时，点击暂停 deck 的 BeatSync 只允许 BPM / grid 间距变化，播放头、viewport 和 stable frame 锚点都不能移动。
- 从轨 / follower 已开启 BeatSync 且暂停时，点击播放必须从吸附到参照 deck grid 的位置开始播放。

### 编辑模式

- 编辑模式播放时，声音也必须稳定连续。
- 放大查看时，波形仍有足够细节，不能像低清预览被强行拉大。
- 已有波形数据的歌曲进入编辑模式后，应能直接查看和编辑波形细节。
- full layout 下中心线、上下形态、播放头和 grid 层级清楚。
- 调整 BPM、beat grid、cue 或 loop 时，标记变化应准确清楚，波形不乱跳。
- 滚轮放大缩小时，当前编辑的鼓点、grid 或 cue 点应留在用户预期位置附近。
- 编辑模式的大波形不能因为追求双轨模式轻量显示，而牺牲深度 zoom 下的清晰度。

### 布局、空态、主题

- 调整窗口大小、Windows 125% / 150% 缩放、双轨 / 编辑模式切换、展开收起侧栏、切换主题，都不能导致波形比例乱跳、当前关注位置带飞或 overlay 错位。
- 歌曲还没有可用波形、正在加载或正在生成波形时，用户仍应能理解这里是大波形区域。
- 空态或加载态中可以保留时间轴结构、中心线、播放头、cue、loop 或可靠 grid 信息，但显示内容只能出现在歌曲真实时长范围内。
- 从空态或加载态过渡到真实波形可以淡入，但不能突然跳、闪、抖，也不能让覆盖层位置变化。
- 亮色主题下，波形区域不能刺眼，RGB 波形不能发白到失去层次。
- 暗色主题下，波形不能糊成一片高饱和色块，grid、cue、loop、播放头不能抢过波形主体。

### 声音优先级

声音是最高优先级体验。无论波形处于加载、滚动、缩放、seek、拖拽、sync、模式切换还是主题切换中，都必须保证：

- 声音连续。
- 声音不爆音。
- 声音不忽快忽慢。
- 声音不因为波形显示暂停、等待或重算。
- 声音和播放头保持可信同步。

可接受的是波形短暂不完整、受控空态或淡入；不可接受的是为了让波形先准备好而延迟声音。

## 推荐手测流程

这套流程按维护者真实手测执行。只测“先联结后播放”或只看静态最终状态，不算完成验收。

1. 选择简单鼓点、短 loop、普通 5-8 分钟歌曲、较长歌曲、cue / loop 较多歌曲各几首。
2. 在双轨模式载入歌曲，确认不播放也能看到可用大波形；载入第一眼检查歌曲起点前是否干净。
3. 点击 overview 跳转多个位置，观察大波形是否快速、稳定出现。
4. 做远距离 seek，确认允许受控加载态或空态加淡入，但不能抽搐、闪烁、跳帧或新旧位置混乱。
5. 播放中在双轨模式连续按 A / D 做固定 4 beats 近距离热键跳转；在编辑模式切换 beat 步长后分别按 A / D 和点击前进 / 后退 beats 按钮。
6. 拖拽大波形，特别是向前回拉，观察是否跟手、流畅、不断层、不卡顿。
7. 重复播放、拖拽、seek 和缩放，确认已有波形数据的歌曲不会表现得像还在现场计算波形。
8. 播放至少 30 秒，观察是否稳定滚动，无持续抖动、抽搐、闪烁、热气感或缩放变化。
9. 检查 grid、cue、loop、播放头和波形是否始终作为一个整体滚动。
10. 在未播放或暂停状态下分别开启联结和 BeatSync，观察上下 deck 网格线是否立即严格对齐。
11. 在至少一条 deck 正在播放时分别点击开启联结和 BeatSync，观察点击当下和随后滚动中的上下 deck 网格线是否立即对齐。
12. 一条 deck 正在播放、另一条 deck 暂停时，点击暂停 deck 的 BeatSync，确认暂停 deck 只改变 BPM / grid 间距，播放头、当前 anchor、viewport 和 stable frame 锚点都不移动。
13. 从轨 / follower 已开启 BeatSync 且暂停时，点击播放，确认播放开始位置已经吸附到参照 deck grid。
14. 在其中一条 deck 接近歌曲开头时开启联结或 BeatSync，确认 0 秒之前的空白区域从开启瞬间开始就没有 grid、节拍刻度、虚拟小节线或假内容。
15. 开启联结后，拖拽任意一条 deck 的大波形，确认两条 deck 像一个整体一样平移。
16. 开启 BeatSync 后，拖拽任意一条 deck 的大波形，确认拖动期间只有当前 deck 跟手移动；松手后当前 deck 就近吸附到参照 deck grid。
17. 联结开启后和 BeatSync 拖拽松手后，检查最终屏幕 x 坐标上是否仍有一到数个像素的旧帧偏移，并确认 follower 小节线没有变成大节线。
18. 开启联结后，使用滚轮连续放大缩小任意一条 deck，确认两条 deck 像一个整体一样缩放。
19. 使用滚轮连续放大缩小波形，确认缩放流畅、不卡顿、不跳级、不闪烁，覆盖层和波形仍然对齐。
20. 边播放边拖拽、seek、缩放或开关联结 / BeatSync，确认组合操作下不闪、不抖、不错位，声音不受影响。
21. 调整窗口大小，展开或收起侧栏，确认波形比例和覆盖层不乱跳。
22. 切到编辑模式，放大查看鼓点和 grid，确认细节足够用于编辑判断。
23. 调整 BPM、beat grid、cue 或 loop，确认标记变化清楚，波形不乱跳。
24. 分别切换亮色和暗色主题，确认波形区域、grid、cue、loop、播放头都清楚。
25. 在高 DPI 或 Windows 125% / 150% 缩放下检查线条、cue 点、loop 和播放头是否清晰。
26. 对照 Rekordbox 同歌同位置参考图或 contact sheet，确认大体形状和鼓点起点接近。

## 一票否决项

出现以下任一用户可感知问题，本次大波形验收不通过：

- 为了波形显示、滚动、缩放、seek、拖拽、联结或 BeatSync，声音出现卡顿、爆音、断续、变速或瞬间静音。
- 鼓点起点明显糊、提前或拖尾。
- 拖拽大波形时出现不可解释的空白、断层、卡顿或旧位置残留。
- 远距离 seek 后出现抽搐、闪烁、来回跳帧或新旧位置混乱。
- 近距离 beat 跳转触发不可解释的空白、空态、淡入、明显重绘闪烁，或跳转距离不一致。
- 播放过程中持续抖动、频闪、热气感、卡顿或缩放变化。
- grid、cue、loop、播放头和波形出现分离、漂移或滚动速度不一致。
- 滚轮放大缩小时出现明显卡顿、跳变、闪烁、当前位置丢失或覆盖层错位。
- 已有波形数据的歌曲在播放、拖拽、seek 或缩放时仍表现出现场计算造成的等待、闪烁、重绘抖动或卡顿。
- 联结或 BeatSync 开启时上下 deck 网格线没有严格对齐，或滚动中出现临时错位、漂移、追赶、抖回。
- 点击开启联结或 BeatSync 时出现明显的先抽动再贴合、两次位置变化、临时错位或追赶感。
- 点击 MASTER / leader deck 的 BeatSync 后，声音没有对应移动但主轨波形右移。
- 一条 deck 播放、另一条 deck 暂停时，点击暂停 deck 的 BeatSync 导致暂停 deck 播放头、viewport、stable frame 锚点移动，或者暂停 deck 做出追播放 deck 的动作。
- 从轨 / follower 已开启 BeatSync 且暂停时，点击播放后没有从吸附到参照 deck grid 的位置开始播放，或播放后才追齐。
- 点击联结或 follower deck 的 BeatSync 后，follower 原本的小节线或普通线被错误重定义成大节线。
- 点击开启联结或 BeatSync 后网格线仍歪，只有拖拽一下大波形才对齐。
- 载入、换歌、缓存恢复、worker 首帧返回或点击联结 / BeatSync 时，歌曲起点前出现 grid、节拍刻度、虚拟小节线、假波形或其它会让用户误判起点前仍有内容的视觉元素。
- 联结开启时，拖拽或缩放任意一条 deck 没有整体作用到双轨。
- BeatSync 开启时，拖拽任意一条 deck 导致另一条 deck 跟随平移，或松手后不是当前 deck 按松手位置就近吸附。
- 只有从轨开着 BeatSync、主轨 / MASTER 没开 BeatSync 时，拖拽主轨后从轨仍保持 BeatSync follower 状态。
- BeatSync 按钮出现闪烁，或 UI 试图用闪烁表达 `tempo-only` / 半锁定 / 第三种 BeatSync 状态。
- BeatSync 拖拽松手后，native transport 或日志显示已经对齐，但最终屏幕上的大波形、grid 或 overlay 仍因旧 stable frame、旧 viewport 或旧 presentation offset 出现偏差。
- 缩放、窗口变化或模式切换后当前关注位置被带飞。
- 连续快速操作时出现卡顿、抽搐、闪烁、错位或新旧画面混乱。
- 播放头被波形、grid、cue 点或 loop 遮挡。
- 高 DPI 或系统缩放下波形、grid、cue、loop、播放头明显发虚或粗细跳变。
- 空态或加载态切到真实波形时出现跳动、闪烁、覆盖层位置变化。
- 播放头、grid、cue、loop 和声音或时间轴明显不同步。
- 亮色或暗色主题任一主题下不可读。
- 编辑模式深度 zoom 下细节明显不足，无法用于节拍编辑判断。
- 需要用户依赖 Rekordbox 才能获得可用大波形体验。
