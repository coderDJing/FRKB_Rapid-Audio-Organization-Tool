# Mixtape 自动录制全局网格重构说明

## 背景

当前 Mixtape 自动录制窗口已经支持编辑 BPM 包络线，但在多首歌重叠时，修改全局 BPM 包络后，重叠区网格线仍然会发生错位。这个问题不是单纯的渲染问题，也不是单纯的吸附问题，而是当前时间模型本身与产品目标不一致。

## 遇到的问题

- 现在的实现会把全局 BPM 包络线投影成每一首歌各自的一份 tempo map。
- 每首歌会根据自己的原始 BPM、网格基准和起点，独立生成自己的网格线。
- 当多首歌发生重叠时，它们虽然共享同一条 BPM 包络线，但并不真正共享同一条全局拍线。
- 结果就是某一根网格线可能可以对上，但前后几根网格线会逐渐漂移。
- 修改 BPM 包络线之后，系统只能通过重新吸附或补偿 `startSec` 来“事后修正”，而不是让所有歌曲天然一起变化。
- 多次编辑 BPM 包络线之后，这种事后修正会不断积累误差，表现为有时还能吸附回去，有时已经吸不回去。

## 理想情况

- BPM 包络线应该先生成一条唯一的全局 master grid。
- 所有歌曲在时间线上看到的网格线，本质上都应该来自这同一条全局网格。
- 两首歌只要在某一时刻是对齐的，那么修改 BPM 包络线后，它们在重叠区仍然应该继续对齐。
- 歌曲应该随着全局 BPM 包络线一起变化，而不是在包络线变化之后再对单首歌做补吸附或补偿。
- 可见网格线、拖拽吸附、播放同步、导出结果，都应该基于同一套全局拍线定义。

## 实现思路

- 引入一条新的全局时间基准，统一描述“绝对时间”和“全局拍位”之间的映射。
- BPM 包络线只负责生成这条全局时间基准，不再直接为每一首歌单独投影 tempo map。
- 每首歌只保留自己的源拍点定义，例如首拍位置、源网格 BPM 和大节偏移。
- 歌曲进入时间线后，不再自己生成独立的全局网格，而是把自己的源拍点贴合到全局 master grid 上。
- 渲染层只从全局 master grid 读取可见网格线，再结合每首歌的源拍点关系来决定轨道上的显示位置。
- 吸附逻辑不再依赖每首歌独立计算的网格，而是直接吸附到当前可见的全局网格线。
- 播放同步和导出阶段也使用同一条全局拍线，保证编辑、预览和输出结果一致。

## 重构目标

- 去掉“全局 BPM 包络线投影到每首歌各自 tempo map”这条旧思路。
- 去掉“修改包络线后再对单首歌做事后补吸附”的补救逻辑。
- 建立“全局 master grid 是唯一真相，歌曲统一贴合这条网格”的新模型。
- 让 Mixtape 自动录制的编辑行为、视觉反馈、吸附结果和播放结果在同一个时间模型下工作。
