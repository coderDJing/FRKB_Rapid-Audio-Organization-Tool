# 双轨播放内核独立化重构目标草案

更新时间：2026-04-20

## 1. 文档定位

这份草案只说明重构目标、边界、阶段和验收标准。

不讨论：

- 具体代码实现
- 具体接口字段
- 具体线程模型细节
- 具体文件修改方案

本草案的作用是先把方向定死，避免后续又回到“继续在页面卡顿上打补丁”的老路。

## 2. 背景

当前双轨模式的核心问题不是单一 bug，而是架构耦合问题：

- 双轨播放体验仍然会受到主窗口页面侧卡顿影响
- 歌曲列表波形生成、BPM 分析、调性分析、网格更新、列表刷新、波形绘制等事件，理论上都可能与双轨模式争抢前端主线程
- 一旦页面侧出现瞬时阻塞，用户感知到的就不只是“界面掉帧”，而可能是“播放也被带着抖”

如果目标是让双轨模式成为稳定可靠的核心能力，那么“声音连续性”和“页面流畅性”必须解耦：

- 页面卡顿可以容忍
- 声音卡顿不能容忍

## 3. 核心判断

双轨模式后续应从“页面驱动的播放”升级为“独立音频引擎驱动的播放”。

也就是说：

- 页面只负责控制和显示
- 播放内核负责时钟、解码、缓冲、混音、输出和同步
- 页面不是播放真相源，只是播放状态的观察者和操作者

## 4. 重构目标

### 4.1 第一目标

在任何页面侧卡顿、列表刷新、波形生成、BPM/调性/网格分析进行中的情况下，双轨模式的声音输出都不应被打断。

### 4.2 第二目标

当页面发生短暂卡顿时，允许界面显示落后于真实播放进度；页面恢复后，应能自动追上当前音频状态，而不是拖着音频一起卡。

### 4.3 第三目标

建立单一可信的播放时钟，使双轨模式中的播放位置、速率、同步状态、主从关系、crossfader、增益等关键状态不再由多个前端链路各自推测。

### 4.4 第四目标

将双轨模式从歌曲列表、波形预览、BPM/调性/网格分析、封面加载、普通播放器 UI 等前端高频更新源中隔离出来，避免非播放事务影响播放稳定性。

### 4.5 第五目标

为后续更复杂的能力预留稳定基础，包括但不限于：

- 更可靠的双轨同步
- 更稳定的 seek / cue / loop 响应
- 更强的容错与恢复能力
- 后续复用到其他播放场景的可能性

## 5. 目标状态

重构完成后，双轨模式应满足以下状态描述：

### 5.1 播放内核独立

双轨播放应由独立音频引擎承担，其运行不依赖 renderer 的渲染节奏，也不应把 UI 主线程当作播放调度核心。

### 5.2 页面角色降级为控制层

页面负责：

- 发出播放控制意图
- 展示当前状态
- 接收快照并驱动 UI

页面不负责：

- 维护最终播放时钟
- 直接决定真实音频输出时序
- 依赖连续渲染来保证声音稳定

### 5.3 状态以快照为准

页面显示的播放位置、同步状态、主从状态、增益、波形游标等，应以播放内核提供的状态快照为准，而不是由多个页面侧局部逻辑各自估算后拼出来。

### 5.4 页面可掉帧但不能拖垮声音

允许出现：

- 页面短暂掉帧
- 波形短暂晚于声音
- 列表或装饰性 UI 延迟刷新

不允许出现：

- 因页面卡顿造成声音停顿
- 因分析完成事件风暴造成双轨声音抖动
- 因普通列表更新影响双轨输出连续性

### 5.5 恢复后自动追平

当页面恢复响应后，应按当前真实播放状态重建显示，而不是从旧的 UI 状态继续缓慢补帧。

## 6. 设计原则

### 6.1 音频优先

所有权衡中，优先保证声音连续性，其次才是视觉连续性。

### 6.2 单一时钟

双轨模式必须只有一个权威播放时钟，所有 UI 都围绕这个时钟显示，而不是每个局部组件各算一套时间。

### 6.3 控制与显示分离

控制命令和显示状态应明确分层，避免“发命令的地方同时也负责推导最终状态”。

### 6.4 非播放事务隔离

分析、波形、列表刷新、封面、普通播放器状态更新等都属于非播放事务，应避免直接干扰双轨播放主链路。

### 6.5 可观测

后续架构必须天然支持定位以下问题：

- 是音频内核卡了
- 还是页面卡了
- 是命令链路堵了
- 还是状态同步延迟了

## 7. 非目标

本次重构的非目标如下：

### 7.1 非目标一

不以“一次性把所有播放相关模块全部推倒重写”为目标。

### 7.2 非目标二

不要求在第一阶段就解决所有视觉卡顿。

第一阶段的重点是：

- 声音独立稳定
- 页面能追上声音

### 7.3 非目标三

不要求这份草案立即确定最终采用“独立线程”“独立进程”还是其他承载形式的全部细节。

这份草案只要求先锁定“必须独立于页面主线程”的方向。

### 7.4 非目标四

不把双轨模式重构范围无上限扩展到全部音频功能。

初始范围应优先围绕双轨模式本身。

## 8. 重构范围

建议纳入范围：

- 双轨模式播放时钟
- 双轨模式音频输出
- 双轨模式 seek / play / pause / cue / loop / sync / gain / crossfader 的权威状态来源
- 页面与播放内核之间的控制与状态同步方式
- 页面卡顿后的状态追平机制

建议暂不纳入第一阶段范围：

- 全量歌曲列表架构重写
- 全量波形系统重写
- 普通单曲播放器整体重构
- Mixtape 全量迁移

## 9. 阶段性目标

### 9.1 第一阶段：播放权威源独立

目标：

- 双轨模式存在明确的独立播放权威源
- 页面不再是播放真相源
- 页面卡顿不应直接导致声音卡顿

### 9.2 第二阶段：状态同步稳定

目标：

- 页面通过快照或受控更新机制显示双轨状态
- 页面恢复后能快速追平真实播放位置
- 双轨组件之间不再各自维护互相冲突的时间推导

### 9.3 第三阶段：非播放事务彻底隔离

目标：

- 波形生成
- BPM/调性/网格分析
- 列表刷新
- 其他页面更新

以上事务在高负载下也不应影响双轨声音连续性。

### 9.4 第四阶段：可靠性与复用性收敛

目标：

- 明确异常恢复策略
- 明确状态重建策略
- 明确后续是否复用到其他播放场景

## 10. 验收标准

重构完成后，至少应满足以下验收标准：

### 10.1 声音连续性

在以下场景中，双轨声音不应出现可感知停顿、断裂、卡顿：

- 歌曲列表波形生成
- BPM 分析完成
- 调性分析完成
- 网格更新完成
- 歌曲列表刷新
- 普通页面高频重绘

### 10.2 页面追平能力

当页面出现短暂掉帧或阻塞后：

- 声音继续
- 页面恢复后，播放头、波形、状态显示应快速追上当前真实进度

### 10.3 控制稳定性

以下操作应保持一致和稳定：

- 播放 / 暂停
- seek
- cue
- loop
- sync
- gain
- crossfader

不能出现“页面状态已变，真实播放没变”或“真实播放已变，页面长期不知情”的混乱状态。

### 10.4 状态单一性

系统内应能明确指出：

- 哪个组件或模块是双轨播放的权威状态源
- 哪些模块只是控制端
- 哪些模块只是显示端

### 10.5 可定位性

出现问题时，应能快速判断是：

- 音频内核问题
- 页面渲染问题
- 状态同步问题
- 控制命令问题

## 11. 风险与取舍

### 11.1 复杂度会上升

播放内核独立后，状态同步、异常恢复、命令时序会比现在复杂。

### 11.2 视觉与声音会被刻意分离

这不是副作用，而是目标的一部分：

- 允许页面短暂落后
- 不允许声音被页面拖死

### 11.3 迁移必须分阶段

如果试图一次性把双轨、普通播放器、Mixtape、波形系统、分析系统一起重做，风险会直接爆炸。

### 11.4 必须接受“先保声音，再修视觉”

这套方向的核心价值在于先守住声音连续性。视觉层的细节平滑、动画一致性、局部体验精修，应排在之后。

## 12. 当前建议结论

当前建议正式确认以下方向：

1. 双轨模式后续采用“独立播放内核”路线，不再把 renderer 优化视为最终解法。
2. renderer 只做控制与显示，不做最终播放时序真相源。
3. 后续所有双轨性能修复，都应服从这一架构目标，避免继续堆积临时补丁。
4. 第一阶段只聚焦双轨模式，不扩大战线。
