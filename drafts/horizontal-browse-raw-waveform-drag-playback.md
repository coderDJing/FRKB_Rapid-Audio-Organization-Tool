# 双轨横推 raw 波形拖动播放行为草案

## 背景

双轨横推模式下，用户拖动 `raw` 细节波形时，如果该轨还在继续正常播放，听感和定位都会发飘。
这类交互更合理的处理是：

- 按下拖动时，立即停止该轨的正常连续播放
- 拖动过程中，只让 UI 跟随手势移动，不继续正常播放
- 松手后，把播放头提交到拖动后的目标位置
- 如果拖动前该轨原本正在播放，则在新位置恢复播放

## 本次落地的务实版本

当前阶段先实现稳定版本，不直接做 scratch 声：

1. 用户按下 `raw` 波形开始拖动时：
   - 记录该 deck 在拖动前是否正在播放
   - 记录拖动前该 deck 是否处于 sync 状态
   - 若拖动前正在播放，则立即暂停该 deck

2. 用户拖动过程中：
   - 只更新细节波形视图位置
   - 不继续正常播放该轨音乐

3. 用户松手时：
   - 将播放位置提交到拖动结束时的锚点位置
   - 若拖动前原本正在播放，则恢复播放
   - 若拖动前原本开启了 sync，则恢复播放前先重新执行一次 beatsync

## 未来目标效果

后续如果要做更像 Rekordbox 的体验，目标不是播放一个固定的刮擦音效，而是：

- 在拖动期间，把当前音乐本身作为声音来源
- 根据手势移动方向和速度，实时前后读取音频采样
- 向前拖时正向变速播放
- 向后拖时反向变速播放
- 小范围来回拖动时，形成类似黑胶搓盘的自然刮擦感

也就是说，未来的 scratch 声应该来自音乐本体，不是预先录好的固定音效。

## 为什么当前先不做 scratch

当前横推双轨主播放链路由 Rust 侧 transport 直接负责输出，已经包含：

- 双 deck 混音
- crossfader 增益
- seek / play / pause
- beat sync / leader 逻辑

而项目里现有的 `scrub` 实现是在 renderer 侧的 beat align 预览链路里，架构层级不同。
如果把 scratch 临时硬拼到横推双轨里，会把播放状态、输出链路、sync 状态和增益控制搞得很容易打架。

所以当前阶段优先选择稳定、可控、容易验证的方案：

- 拖动时暂停
- 松手后 seek
- 必要时恢复播放并重新对齐 sync

## 验收要点

- 正在播放的 deck，按下 `raw` 波形拖动时，声音应立即停止正常连续播放
- 拖动未松手期间，不应继续听到该轨正常推进
- 松手后，播放头应跳到拖动后的目标位置
- 若拖动前正在播放，则松手后应继续播放
- 若拖动前未播放，则松手后不应自动开始播放
- 若拖动前启用了 sync，则恢复播放前应重新对齐节拍
