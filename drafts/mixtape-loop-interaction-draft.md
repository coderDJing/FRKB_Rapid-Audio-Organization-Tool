# Mixtape Loop 交互草案

## 文档目标
- 这份文档只讨论 `Loop` 的交互方案
- 先把已经达成一致的规则写死
- 剩余未定项尽量收敛到最少

---

## 当前共识

### 1. 场景前提
- 当前按 DJ 场景处理
- 同一首歌允许存在多个 loop
- 但多个 loop 之间严格不允许重叠
- 当前版本仍不考虑“自重叠 / 嵌套 loop”玩法

### 2. 已有 Loop 时，开始位置的选线规则
针对“已经有一个或多个 Loop 的轨道”，新的开始位置候选网格线这样限制：

- 不允许选中任意已有 loop 占用区间内部的任何网格线作为开始位置
- 允许选中任意已有 loop 的最右边界线作为开始位置
- 允许选中所有不属于任意已有 loop 占用区间的网格线作为开始位置

换句话说：

- 开始位置不能落在任何已有 loop 占用区间内部
- 唯一例外是：某个已有 loop 的最右边界线可以作为新的开始位置

### 3. A/B 的定义方式
- 用户手动选择两根合法网格线
- 系统自动把两根线里靠前的一根认定为开始位置
- 系统自动把两根线里靠后的一根认定为结束位置
- 两个点只能在同一首歌内产生
- 不允许第一个点在一首歌里、第二个点在另一首歌里

换句话说：

- `start = min(line1, line2)`
- `end = max(line1, line2)`

### 4. 两根线的选择约束
- 用户进入 `Loop` 参数模式后，直接就可以点击合法网格线开始选区
- 第一次合法点击落在哪一首歌上，这首歌就成为当前选区目标轨道
- 在第 2 根线落下之前，下一次点击只能继续在这首歌里选择别的合法网格线
- 其它轨道上的网格线在此时直接进入不可选状态
- 如果用户要放弃这次选择，应有明确取消方式，例如 `Esc`
- 第 2 根线不能落在任何已有 loop 占用区间内部
- 两根线确定的新区间也不能与任何已有 loop 区间相交

### 5. 重复次数的调整方式
- `repeatCount` 继续通过波形内的 `- / +` 控件调整
- 不再通过拖拽重复段尾部来改 `repeatCount`

### 6. 已有 Loop 后的处理方式
- 当前版本不支持直接重设某一段 loop 的开始/结束位置
- 如果用户想改某一段 loop 的开始/结束位置，只能先清除该段，再重新选择两根线
- 当前版本 Loop 编辑动作保留：
  - 选两根线新建 loop
  - 调整某一段 loop 的 `repeatCount`
  - 清除某一段 loop

### 7. 交互拆分原则
- 开始/结束位置选择和 `repeatCount` 调整必须分离
- 每条交互链只做一件事
- 不允许再出现“一次手势同时猜开始/结束位置和重复次数”的逻辑

---

## 当前推荐模型

### 方案主线
1. 用户进入 `Loop` 参数模式
2. 用户直接点击第一根合法网格线
3. 系统锁定当前轨道为选区目标轨道
4. 用户继续在同一首歌里点击第二根合法网格线
5. 系统自动按前后顺序计算开始/结束位置
6. 如果新区间不与已有 loop 相交，则新增一段 loop
7. 用户点击任意 loop 段，显示该段 loop 的小工具条
8. 后续只通过 `- / +` 改这段 loop 的重复次数，或通过 `清除` 删除这段 loop

### 为什么先这么做
- 交互单义，用户更容易理解
- 符合 DJ 场景，不放大罕见玩法
- 实现成本和后续维护成本都更低

## 需要继续讨论的问题

### 1. 第 1 根线选中后的状态反馈
当前建议直接定成这一版：

#### 进入条件
- 进入 `Loop` 参数模式后，直接就可以点击第一根合法网格线
- 不需要额外点“设 A”或“开始选区”

#### 第 1 根线选中后，界面反馈
- 当前轨道进入高亮状态
- 第 1 根线保持高亮，作为当前锚点
- 当前轨道顶部显示一个短胶囊提示：
  - 主文案：`已选第1点`
  - 辅助文案：`继续在本轨选第2点`
  - 取消提示：`Esc 取消`
- 其它轨道直接进入禁用状态
  - 其它轨道网格线不再响应 hover
  - 其它轨道点击无效
  - 其它轨道只做轻微灰化，不额外塞文字

#### 点错其它轨道时
- 默认直接禁用，不触发跨轨选择
- 可以考虑给一次非常轻的提示：
  - `请先完成当前轨道的第2点`

这套方案的目标是：
- 让用户明确知道“当前锁定的是哪首歌”
- 明确知道“下一步还差第 2 根线”
- 不让提示扩散到整个时间线，避免界面变脏

### 2. 删除 Loop 的入口放哪里
当前已确认：

#### 主入口
- 当前选中的 loop 显示一个明确的 `清除` / `删除 Loop` 按钮
- 删除按钮应跟“当前选中的 loop 段”绑定，而不是跟整首歌绑定
- 点击任意 loop 段，都可以让该段 loop 成为“当前选中的 loop”

#### 当前版本不做
- 当前版本不支持通过 `Delete / Backspace` 删除 loop

为什么这样定：
- 删除是针对某一个 loop 实体的动作
- 如果未来同一首歌允许多段 loop，删除入口必须能区分“删哪一段”
- 当前版本先不引入键盘焦点和快捷删除逻辑，避免状态变复杂

### 3. `- / +` 控件放哪里
当前已确认：

#### 不推荐的方案
- 不推荐固定放在歌曲波形左上角
  - 同一首歌将来如果允许多段 loop，会互相打架
- 不推荐强行塞进 loop 段内部
  - 段很小时根本放不下
  - 视觉也容易挡住波形

#### 当前推荐方案
- `- / 重复 xN / + / 清除` 做成“当前选中 loop 的浮动工具条”
- 工具条跟着“当前选中的 loop 段”走
- 但工具条本身不依赖 loop 段内部空间
- 点击任意 loop 段，都可以显示这套工具条

#### 已确认的摆放规则
1. 默认显示在当前 loop 上方
2. 水平对齐到当前选中 loop 段的中点
3. 工具条位置要做左右边界钳制，不能飞出轨道可视区
4. 工具条永远悬浮在 loop 段外部，不塞进 loop 段内部
5. 同一时刻只允许一个 loop 段显示这套工具条

这样即使未来一首歌有多段 loop：
- 也只会有“当前选中的那一段”显示工具条
- 不会出现多个 `+/-` 按钮重叠

### 4. 用户点到非法网格线时如何反馈
当前建议直接定成“轻反馈，但绝不静默失败”。

#### 总原则
- 不弹打断式对话框
- 不给整屏大提示
- 反馈尽量贴近当前正在编辑的那条轨道
- 但用户点错时，必须知道“为什么没生效”

#### 非法场景 A：第 1 根线选中后，用户去点别的轨道
处理方式：
- 其它轨道本身已经是禁用态，不响应正常 hover
- 如果用户仍然去点击其它轨道：
  - 不切换目标轨道
  - 当前锁定轨道和第 1 根线做一次轻微闪烁
  - 当前轨道顶部短提示一次：
    - `第2点只能选在当前轨道`

#### 非法场景 B：用户点击了已有 loop 占用区内不允许作为开始位置的线
处理方式：
- 这些非法线在 hover 时直接弱化显示
- 不显示正常的 `开始位置` / `结束位置` 标签
- 点击后，在当前轨道顶部短提示一次：
  - `Loop 区内不能作为起点`

#### 非法场景 C：用户第二次又点到了同一根线
处理方式：
- 不生成 Loop
- 当前已选中的第 1 根线轻微闪烁一次
- 当前轨道顶部短提示一次：
  - `请再选另一根线`

#### 非法场景 D：用户选出的新区间会与已有 loop 相交
处理方式：
- 不新增 loop
- 当前选中的第一根线轻微闪烁一次
- 当前轨道顶部短提示一次：
  - `不能与已有 Loop 重叠`

#### 非法反馈的显示位置
- 统一显示在“当前锁定轨道”的顶部短胶囊区域
- 不在别的轨道上额外塞提示文字
- 不做全局 toast

为什么这样定：
- 反馈离用户视线近
- 不会把整条时间线搞脏
- 用户能明确知道当前失败原因

## 当前不做的内容
- 不支持任意重叠 loop
- 不讨论“多层 loop 嵌套”
- 当前版本不支持直接重设 `A/B`
- 当前版本不支持通过 `Delete / Backspace` 删除 loop
- 不讨论非常实验性的 DJ 玩法

## 当前选中 Loop 的视觉状态

### 目标
- 用户一眼就能看出“当前正在操作的是哪一个 loop”
- 不把整个波形盖脏
- source / repeat 要能区分，但不能太花

### 当前推荐方案

#### 1. 当前选中的 loop
- 整个 loop 区间轻微提亮
- 外边界线更亮、更实
- 小工具条出现时，这一段保持选中态
- 提亮强度保持克制，不做强烈荧光感

推荐理解：
- 选中态应该像“被聚焦的编辑对象”
- 不是警告态，也不是高饱和强调态

#### 2. 未选中的 loop
- 保持可见
- 但整体比当前选中的 loop 略暗一档
- 不显示工具条
- 当前确定：未选中的 loop 采用整体降一点亮度的方式退后

这样可以避免：
- 时间线上出现多个 loop 时，不知道当前操作的是哪一个

#### 3. source / repeat 的区分方式
- `loop-source` 和 `loop-repeat` 继续保留区分
- 但区分应尽量克制
- 当前倾向：弱纹理区分，而不是明显色差区分

建议：
- `loop-source`
  - 更实一些
  - 边界更明确
- `loop-repeat`
  - 轻一点
  - 保留弱纹理或弱条纹
  - 不使用明显跳色

目标：
- 用户知道哪段是原始段，哪段是重复段
- 但不会把波形本体压没

#### 4. 当前选中 loop 与其它 UI 的关系
- 当前选中 loop 的高亮层级应低于工具条
- 高于普通未选中 loop
- 不应该压过波形细节太多

换句话说：
- 工具条最显眼
- 当前选中 loop 第二显眼
- 其它 loop 退后

### 当前不推荐的方案
- 不推荐给当前 loop 上很重的整块纯色蒙层
- 不推荐 source / repeat 用两套差异很大的颜色
- 不推荐未选中 loop 完全隐身
- 不推荐只靠工具条判断当前选中对象

### 待进一步决定的细节
- 当前没有新增待确认项
