# 四拍网格工程级重构草案

创建日期：2026-07-14
状态：已开始实施；每完成一个可验证条目立即勾选，禁止提交、推送或发布

## 1. 文档目的

本文档用于定义 FRKB 全工程的网格语义重构。需求通过“一次一个问题”的方式逐项确认；每个问题都记录用户选择、推荐方案和最终结论。当前已补齐初轮代码影响面、数据迁移、实施顺序和验收标准。

本文档同时作为本轮实施与验收清单。

问答规则补充：只有会改变当前实现、存在真实产品分叉，或代码事实无法替用户决定的事项才向用户提问。推荐方案与当前实现完全一致时，直接在代码审计后记录为“继承现状”，不得浪费一次用户问答。

## 2. 已确认的产品目标

- 本次不是单纯修改波形显示，而是覆盖整个工程的网格语义、数据模型、分析链路和所有消费者。
- 保留每一拍的普通细线。
- 保留正确的四拍线；四拍线必须由正确的 downbeat 相位生成。
- 删除大节线、小节线以及依赖这些层级概念的工程实现。
- 后续需求确认严格一次只讨论一个问题，每个问题必须提供推荐选项和推荐理由。

## 3. 已确认的基础网格语义

在后续问题没有推翻本节之前，基础网格暂定只包含两种线：

```text
downbeat  beat  beat  beat  downbeat  beat  beat  beat
    |      ·     ·     ·        |      ·     ·     ·
```

- `beat`：每一拍的普通细线。
- `downbeat`：每四拍一次、表示小节第一拍的统一四拍线。
- 不允许在基础网格中继续派生 8 小节、16 小节、32 拍等大小层级线。

## 4. 工程改造范围（待逐项核实）

最终影响面至少包括：

- 自动网格分析、候选生成、候选选择和置信度。
- 固定 BPM 网格与动态多 clip 网格的数据模型。
- 歌曲持久化字段、缓存签名、失效判断和旧数据迁移。
- main / preload / renderer IPC 与共享 TypeScript 类型。
- 主播放器、双轨、编辑模式和各种 waveform overlay。
- Mixtape 时间映射、网格渲染、吸附、拖动、loop、tempo segment 和 transport。
- BeatSync、节拍器、循环和所有依赖逐拍位置的功能。
- 不改写旧段落算法或展示；但必须阻断旧段落分析的生产链路并保留历史结果。
- Rekordbox 导入、truth、benchmark、错误分类和诊断脚本。
- 单元测试、真实样本回归、文档和兼容清理。

本节目前只是范围清单，不代表已经完成全量代码审计。

## 5. 需求决策记录

### 问题 1：段落分析是否继续拥有独立的 phrase / section 边界

状态：已确认。

用户决定：本次四拍网格工程重构不处理段落分析算法或展示；旧生产链路立即阻断，现有结果原样保留，段落分析功能后续单独立项改写。

当前草案采用以下边界：

- 保留旧段落算法源码与历史结果，不在本次顺手重写或删除；但停止旧段落任务入队、worker 执行和结果写回，以免两个网格语义混用。
- 新两层网格不向段落分析提供伪造的 8 / 16 / 32 小节兼容层级。
- 段落分析后续必须基于新网格语义单独重新设计，不能依赖本次临时兼容 fallback。

### 问题 2：新的四拍相位字段是否彻底取代旧 `barBeatOffset`

状态：已确认。

用户选择：A。

最终决定：活动代码、共享类型、IPC、持久化事实源和动态网格 clip 统一改用 `downbeatBeatOffset`，其合法值只能表达 `0..3` 的四拍 downbeat 相位。旧 `barBeatOffset` 不允许作为长期活动字段继续存在。

### 旧数据库迁移的已查明事实

- 当前 SQLite 主库通过 `PRAGMA user_version` 管理 schema，现有版本为 `35`。
- 网格字段不是独立 SQLite 列，主要存放在 JSON blob 中，因此不能只靠 `ALTER TABLE` 完成迁移。
- 已确认至少需要处理 `song_cache.info_json`、`mixtape_items.info_json`、`set_items.analysis_json`。
- 固定网格在 JSON 根对象保存旧字段；动态多 clip 网格还在 `beatGridMap.clips[*]` 中保存旧字段。
- `SongBeatGridMap.signature` 包含旧字段值和字段语义；迁移后必须升级 map version 并重新计算签名。
- 依赖旧签名、旧大小节层级或旧结构语义的段落结果不属于本次迁移和验收范围，后续段落重写时单独处理。
- 还需要继续审计 Rekordbox 快照、benchmark truth、外部分析缓存、Mixtape project snapshot 和独立导入/导出文件，禁止只迁主歌曲缓存。

### 暂定迁移不变量

- 有效旧值迁移公式固定为 `downbeatBeatOffset = ((barBeatOffset % 4) + 4) % 4`，只保留真实四拍相位，明确丢弃 32 拍层级序号。
- 动态网格必须逐 clip 迁移，不能只迁第一段兼容投影。
- 缺失、非有限或无法证明有效的旧值不得静默写成 `0`；应把对应网格标记为不完整并进入明确的重新分析/人工处理路径。
- 迁移完成后活动 JSON 中必须删除旧字段，禁止新旧字段双写。
- schema 升级必须有迁移前一致性备份、磁盘空间检查、SQLite 事务、迁移后计数与语义校验；任一步失败都不得提升 `user_version`。
- 可再生缓存优先失效或清理，不为了保缓存引入双语义兼容层。

### 问题 3：升级后的数据库是否还必须允许旧版 FRKB 直接打开

状态：已确认。

用户选择：A，采用单向 schema 升级。新版本负责迁移旧数据库；升级后的新语义数据库不承诺继续兼容旧版 FRKB。降级必须依赖迁移前备份，不允许新旧字段双写。

### “禁止旧版打开”的现实边界

- 已经发布出去的旧程序不认识未来新增的 schema 标记，新版本无法隔空修改旧程序行为。
- 当前旧代码只在 `user_version < SCHEMA_VERSION` 时迁移，没有在数据库版本更高时拒绝打开。
- 当前 NSIS 配置没有现成的 downgrade 阻断规则；即使新增安装器降级保护，也无法阻止用户手工运行旧 portable / 旧安装目录中的可执行文件。
- 因此，同一个数据库文件、同一个文件路径下，只靠 `user_version`、manifest 或提示文字无法做到绝对禁止旧程序打开。
- 新版本必须增加“数据库版本高于当前程序支持版本时拒绝打开”，但这个保护只能约束本版本及未来版本，不能追溯约束已经发布的旧程序。

### 问题 4：如何阻断旧程序打开未来版本数据库

状态：已确认。

用户决定：不处理当前版本以前已经发布的软件；当前尚无需要兼容的实际用户。从本次版本开始建立数据库 schema 前向兼容防线。

正式规则：

- 每个 FRKB 版本必须声明自己支持的最高数据库 schema 版本，例如 `MAX_SUPPORTED_DATABASE_SCHEMA_VERSION`。
- 打开活动库、切换库、打开独立库副本或准备执行迁移时，第一步只读取 `PRAGMA user_version`。
- 当数据库版本高于当前软件支持上限时，必须立即关闭连接并阻断；禁止执行建表、迁移、查询业务数据、后台扫描、缓存维护或任何写入。
- 阻断提示必须明确说明“数据库由更高版本 FRKB 创建或升级，当前软件无法安全打开，请更新 FRKB”。
- 当数据库版本低于或等于当前软件支持上限时，才允许执行当前版本声明支持的顺序迁移。
- schema 迁移失败时不得提升 `user_version`，不得带病进入业务流程。
- 当前活动库初始化、数据库初始化窗口、库合并来源库、隔离 snapshot 升级等所有数据库入口必须复用同一版本守卫，禁止各写一套判断。
- 本规则从当前版本开始保证未来版本安全；不为更早已发布且没有守卫的旧程序设计额外兼容层。

当前代码审计结论：库合并链路已经会拒绝高于目标版本的来源库，但活动库 `createDatabase()` 目前缺少 `userVersion > SCHEMA_VERSION` 的拒绝分支；本次工程改造必须补齐并统一这个边界。

### 问题 5：固定网格与动态多 clip 网格是否采用完全相同的两层线语义

状态：已确认。

用户选择：A。

最终决定：固定网格与动态多 clip 网格采用完全相同的基础语义。每个网格段只包含逐拍时钟与四拍 downbeat 相位；动态网格相对固定网格唯一增加的能力，是允许歌曲途中出现新的 clip，并在 clip 边界改变 BPM 或拍点相位。

新的动态 clip 数据模型暂定为：

```ts
type SongBeatGridClip = {
  startSec: number
  anchorSec: number
  bpm: number
  downbeatBeatOffset: number // 只允许 0..3
}
```

统一规则：

- 固定网格在逻辑上等价于单 clip，但是否实际落库为 map 由后续问题决定。
- 固定与动态 runtime 都只能生成 `downbeat` 和 `beat` 两种线。
- clip 边界不是大节线、小节线或段落线，不得使用特殊网格层级渲染。
- 动态 clip 不允许保存或恢复任何 8 / 16 / 32 小节序号。
- BeatSync、吸附、loop、transport 和 waveform 必须消费同一个统一 runtime，不允许为固定/动态各保留一套层级判断。

### 问题 6：旧歌曲的有效网格是确定性迁移还是强制重新分析

状态：已确认。

用户选择：A。

最终决定：所有能够证明有效的旧网格均执行确定性迁移，不因 schema 升级重新运行音频分析。

迁移规则：

- 保留原有 BPM、`firstBeatMs` / clip `anchorSec`、`startSec`、`timeBasisOffsetMs`、人工/分析来源和分析算法版本。
- 固定网格使用旧值 `% 4` 生成新的 `downbeatBeatOffset`。
- 动态多 clip 网格逐 clip 转换，禁止只迁移第一段投影。
- 升级 `SongBeatGridMap` schema version，并按新字段与新语义重新计算 signature。
- schema 迁移不得伪装成新算法重分析，不得擅自把旧 `beatGridAlgorithmVersion` 改成当前算法版本。
- 人工修正与自动分析结果采用相同的字段转换，但必须保留来源，禁止迁移后丢失人工优先级。
- 缺失、非法、解析失败或无法验证的网格进入明确的不完整状态；禁止默认补 `downbeatBeatOffset = 0`。
- 旧段落结果不参与网格迁移正确性判断，也不得因本次迁移触发段落重分析。

### 问题 7：固定歌曲是否也统一持久化为单 clip `SongBeatGridMap`

状态：已确认。

用户选择：A。

最终决定：所有有效歌曲网格统一以 `SongBeatGridMap` 作为唯一持久化事实源。固定歌曲保存一个 clip，动态歌曲保存多个 clip。

统一事实源规则：

- 根级 `bpm`、`firstBeatMs`、`downbeatBeatOffset` 不再作为独立事实重复持久化。
- 列表展示、排序、过滤、IPC 简化载荷等需要固定摘要时，只能从 map 计算即时只读投影。
- 只读投影不得被回写成第二份持久化数据，不得参与“哪个值更新”的优先级竞争。
- 固定歌曲的单 clip 必须 `startSec = 0`；其 `anchorSec`、`bpm` 和 `downbeatBeatOffset` 定义整首固定网格。
- 动态歌曲只是在同一个 map 中拥有多个连续 clip，不再走另一套数据结构。
- 所有 active consumer 必须通过统一 runtime 获取拍线、局部 BPM、downbeat 和 clip 查询结果。
- 迁移后活动 JSON 中原有根级网格字段必须删除，禁止为了兼容旧组件继续双写。

### 问题 8：历史 Rekordbox truth、benchmark 和封存快照如何升级 schema

状态：已确认。

用户选择：A。

最终决定：活动数据库、当前维护输入和生产快照迁移到新 schema；已经封存的历史证据保持原文件字节不变，通过可追溯转换生成 v2 派生副本。

历史证据迁移规则：

- 生产代码、活动 benchmark 和新 truth 只接受 v2 schema，不保留运行时 legacy fallback。
- 历史原件不得原地改写、重排或重新格式化，避免破坏已有 SHA、roster、solver 身份和报告绑定关系。
- v2 派生副本必须记录源文件路径/稳定标识、源文件 SHA-256、源 schema 版本、转换器版本和生成时间。
- 转换过程只执行字段语义迁移和必要的 map/signature 重建，禁止借机修改 truth、分类或历史 solver 输出。
- 派生副本必须与原件建立一对一 provenance 关系，任何无法完成身份校验的原件必须 fail closed。
- 历史报告继续代表当时的算法与指标；如果需要按新指标重放，必须生成新的 replay 报告，禁止覆盖旧报告。
- legacy importer 只能存在于明确的离线转换入口，转换结果进入 v2 后，后续链路不得再次感知旧字段。

### 问题 9：逐拍线和四拍 downbeat 线在缩放时如何显隐

状态：已确认。

用户选择：A。

最终决定：网格数据始终保留完整逐拍位置，但渲染层根据相邻拍线的屏幕像素距离统一控制普通逐拍线显隐。

显示规则：

- 四拍 downbeat 线在网格显示开启时保持可见。
- 普通逐拍线只有在相邻拍线达到可辨识的最小像素间距时才绘制。
- 整首歌曲被压缩到较窄波形区域、逐拍线接近粘连时，只显示四拍 downbeat 线。
- 放大波形后普通逐拍线自动恢复，不需要用户切换另一套网格模式。
- 显隐判断必须由共享 helper 统一计算，主播放器、双轨、编辑模式、Mixtape 和 worker 禁止各自硬编码不同阈值。
- 显隐只属于渲染裁剪，不得影响吸附、BeatSync、loop、transport 或任何内部逐拍计算。

### 问题 10：四拍线与普通逐拍线采用什么视觉层级

状态：已确认。

用户最终决定：直接沿用工程中现有四拍 `beat4` 线的视觉，不改成红色，也不重新设计颜色、宽度或透明度。

最终规则：

- 当前 `beat4` 线在新语义中直接更名为 `downbeat` 线。
- 各现有界面的四拍线颜色、宽度、透明度和基本绘制效果保持不变。
- 普通逐拍 `beat` 线的现有视觉保持不变。
- 四拍 `downbeat` 线在普通逐拍线基础上增加朝轨道外侧的突出段，用于不依赖颜色识别四拍起点。
- 双轨上轨的突出段向上延伸，双轨下轨的突出段向下延伸；普通逐拍线不突出。
- 突出段只属于视觉绘制，不得改变拍线真实时间、吸附位置、命中范围或 transport 计算。
- 删除当前 32 拍 `bar` 大线及其专用颜色、宽度、主题变量和绘制分支。
- 禁止每隔 8 / 16 / 32 小节重新加粗、变色或产生第三种网格视觉层级。
- 本次视觉改造的目标是删除多余层级，不是重新设计现有四拍线。

### 问题 10.1：四拍线突出段的尺寸和单轨方向

状态：已确认。

用户选择：A。

最终规则：

- 四拍 downbeat 线的目标突出长度固定为 `3 CSS px`。
- 双轨上轨和所有单轨波形向上突出，双轨下轨向下突出。
- 正常波形区域必须绘制完整 3px，不随歌曲振幅、横向缩放或播放状态变化。
- 只有轨道外侧实际可用绘制空间不足 3px 时，才将突出长度限制为当前可用空间，避免 Canvas 或容器裁切。
- 可用空间限制必须是纯布局保护，不允许按波形高度比例产生不同视觉规格。
- 高 DPI Canvas 必须先按 CSS px 定义，再按设备像素比换算，禁止直接写死物理像素。

### 问题 11：原来依赖 32 拍大线的吸附和编辑操作如何处理

状态：已确认。

用户选择：A。

最终规则：

- 原来以 `bar` 或 `beat4` 为候选的吸附、拖动和编辑操作统一改为四拍 `downbeat` 吸附。
- 原来明确支持逐拍精度的操作继续吸附到每一根 `beat`，不得因为层级删除而降级到四拍精度。
- loop 长度继续按 beat 数计算，4 / 8 / 16 / 32 beat 只是长度，不得重新解释为网格线层级。
- 删除所有不可见的 32 拍吸附点、优先级、搜索分支和快捷路径。
- clip 边界继续作为动态网格编辑对象存在，但它不是大小节线；只有明确的 clip 边界编辑操作才能命中它。
- UI 显隐不能改变吸附数据：普通逐拍线因像素密度暂时不绘制时，逐拍吸附仍然有效。

### 问题 12：新 benchmark 的严格 pass 是否继续要求 downbeat 正确

状态：已确认。

用户选择：A。

最终规则：

- 严格 pass 必须同时满足 BPM/节拍间隔、逐拍毫秒时相、长期网格漂移和四拍 downbeat 相位要求。
- downbeat 错一拍仍然属于失败，禁止通过删除错误分类虚增准确率。
- 删除 `barBeatOffsetMatchedExact32`、32 拍 ordinal、phrase line 等活动评测指标和候选选择依赖。
- 新 truth 使用统一 `SongBeatGridMap` schema；固定歌曲以单 clip 表达。
- candidate oracle、selected result、错误分类和置信度报告继续保留，但全部基于新两层网格语义。
- 历史 benchmark 报告保持原指标原文件不变；按新 schema 重放时必须生成独立 replay 报告。
- 新报告必须显式记录 grid schema version、solver identity、truth identity 和严格阈值，禁止混报新旧口径。

### 问题 13：人工修正网格是否允许被自动分析覆盖

状态：已确认。

- 算法准确率优化继续聚焦正确候选选择，不得通过减少严格 downbeat 检查伪造收益。

### 问题 25：同一首歌的网格是否允许复制到多个业务表

- `mixtape_items.info_json` 还会复制 BPM、首拍、offset、map 等分析字段。
- `set_items.analysis_json` 也保存分析快照。
- `loadSharedSongGridDefinition()` 会从歌曲缓存和 Mixtape item 中 merge 网格，`upsertMixtapeItemGridByFilePath()` 又会按文件路径同步复制值。

用户选择：A。

最终规则：

- 每个物理音频文件只允许存在一份 canonical `SongBeatGridMap`。
- Mixtape item 和 Set item 不再复制 BPM、首拍、downbeat offset 或 map。
- Mixtape / Set 只保存项目自身参数、文件引用和确有必要的项目局部状态。
- `originalBpm`、播放 tempo、gain、转场、loop 等项目参数不等于歌曲网格，可以继续项目化保存。
- 所有页面通过统一网格服务按文件身份读取 canonical map，并通过统一事件接收更新。
- 删除从多个 JSON merge 网格、按文件路径向多个 item 回写 map、以及副本优先级仲裁代码。
- 数据库迁移必须在删除副本前先完成 canonical map 归并和冲突检测。

### 问题 26：canonical `SongBeatGridMap` 存在独立权威表还是分析缓存 JSON

状态：已确认。

代码审计事实：当前歌曲网格主要埋在 `song_cache.info_json` / external analysis cache 中；这些表同时承载大量可再生分析缓存。用户随后明确网格应跟随“完整歌曲分析缓存”的生命周期，而不是脱离分析缓存永久保存。

待确认的核心边界：新 schema 是否建立专用 SQLite 表保存唯一 map，还是继续把唯一 map 放在分析缓存 JSON 中。

用户选择：B。

用户补充的产品语义：网格属于可重新生成的歌曲分析数据。人工调整在正常使用期间受保护，但用户确认完整重新分析、真正清除/重建歌曲分析缓存或歌曲被移除后，旧网格应当被删除；以后重新分析时恢复为当前分析器生成的初始结果。

最终规则：

- canonical `SongBeatGridMap` 继续保存在统一歌曲分析缓存中，不新建脱离分析生命周期的永久网格表。
- 资料库歌曲使用 `song_cache.info_json`，受支持的外部歌曲使用对应 external analysis cache。
- Mixtape 和 Set 不再复制 map，只通过统一服务读取歌曲分析缓存中的 canonical map。
- 正常后台扫描不得覆盖有效人工 map；用户确认完整重新分析后，允许清除人工 map 并生成新的自动分析单 clip map。
- 真正删除歌曲分析缓存、移除歌曲或清理已失效文件时，对应 map 随分析数据一起删除。
- 以后重新加入或重新分析歌曲时不恢复旧人工 map，而是生成新的分析初始结果。
- 普通 waveform 子缓存失效不得误删整份歌曲分析缓存；必须区分“只重建波形”和“完整重新分析”。

### 问题 27：迁移时发现同一歌曲的多个旧网格副本冲突如何处理

状态：已确认。

代码审计事实：当前歌曲缓存、Mixtape item 和 Set item 可能都保存旧网格，且读取链路存在 merge。迁移为单一 canonical map 时，多个副本可能内容不同，也可能缺少可靠更新时间，不能简单按“最后一个”判断。

用户选择：B。

最终规则：

- 数据库迁移时只认歌曲分析缓存中的网格：资料库歌曲使用 `song_cache.info_json`，受支持的外部歌曲使用 external analysis cache。
- Mixtape item、Set item 和项目快照中的旧网格副本一律不参与 canonical map 仲裁，即使副本标记为人工来源。
- 歌曲分析缓存中的有效旧网格按问题 6 的确定性规则迁移为新 map。
- 歌曲分析缓存缺失、损坏或无法迁移时，不从 Mixtape / Set 副本恢复；该曲标记为待重新分析。
- 迁移完成后删除 Mixtape / Set 中所有网格副本字段和同步/merge 逻辑。
- 该选择接受旧业务副本中可能存在的不同人工调整被丢弃，换取明确、可验证的单一迁移来源。

## 8. 目标数据模型

### 8.1 Map schema v2

```ts
export const SONG_BEAT_GRID_MAP_VERSION = 2

export type DownbeatBeatOffset = 0 | 1 | 2 | 3

export type SongBeatGridClip = {
  startSec: number
  anchorSec: number
  bpm: number
  downbeatBeatOffset: DownbeatBeatOffset
}

export type SongBeatGridMap = {
  version: 2
  source: 'analysis' | 'manual'
  clips: SongBeatGridClip[]
  signature: string
}
```

约束：

- fixed song：恰好一个 clip，且 `startSec = 0`。
- dynamic song：两个以上连续 clip，仍使用同一 clip 结构。
- 自动分析只生成单 clip map。
- 用户修改任一 clip 后，整个 map 的 `source` 变为 `manual`。
- `beatGridAlgorithmVersion` 可以继续作为分析 provenance 存在于歌曲分析信息中，但不得成为第二份几何事实源。
- `timeBasisOffsetMs` 继续作为分析坐标 provenance；拍线几何只认 map 中的 clip。

### 8.2 Runtime schema

```ts
export type SongBeatGridLineLevel = 'downbeat' | 'beat'

export type SongBeatGridLine = {
  sec: number
  clipIndex: number
  beatOrdinal: number
  clipBeatIndex: number
  bpm: number
  level: SongBeatGridLineLevel
}
```

判定公式：

```text
shifted = clipBeatIndex - downbeatBeatOffset
shifted mod 4 == 0  -> downbeat
其他                -> beat
```

`clipBoundaries` 继续作为 runtime 的独立数组存在，不进入 `SongBeatGridLineLevel`。

### 8.3 只读固定投影

列表、排序或旧 UI 需要固定摘要时，可以即时计算：

- `bpm`：第一 clip BPM。
- `firstBeatMs`：第一 clip 在 `0` 之后的第一根逐拍线。
- `downbeatBeatOffset`：第一 clip offset。

投影不得持久化、不得回写 map、不得参与 merge。

## 9. 数据库 schema v35 -> v38 迁移设计

说明：四拍网格语义迁移本身落在 v36；随后已落地的时间基准修复继续将正式 schema
提升到 v38。因此当前程序的实际目标与最高支持版本均为 v38，旧 v35 库会按 v35→v36→v38
顺序完成迁移。

### 9.1 前向版本守卫

建议本次提升为：

```ts
const SCHEMA_VERSION = 38
const MAX_SUPPORTED_DATABASE_SCHEMA_VERSION = 38
```

所有数据库入口必须先读取 `PRAGMA user_version`：

- `user_version > 38`：立即关闭连接，阻断并提示更新 FRKB。
- `user_version === 38`：正常打开。
- `user_version < 38`：进入阻断式迁移界面。

守卫必须覆盖活动库、数据库初始化窗口、切库、独立 snapshot 升级和 library merge 来源检查。

### 9.2 迁移前阶段

1. 禁止创建主窗口和启动所有后台任务。
2. 计算源库逻辑大小、WAL、临时事务页和一致性备份所需空间。
3. 空间不足或无法读取空间时阻断，显示明确缺口。
4. 创建临时一致性备份并记录源 schema、大小、SHA-256。
5. 进入不可取消的正式迁移阶段。

### 9.3 canonical song cache 转换

对 `song_cache.info_json` 和受支持的 external analysis cache：

1. 如果存在合法旧 `beatGridMap`，以 map 为准逐 clip 转换。
2. 否则，如果根级 `bpm / firstBeatMs / barBeatOffset` 完整合法，创建单 clip map。
3. offset 使用 `((old % 4) + 4) % 4`。
4. 保留原 `beatGridSource`，生成 map `source`。
5. 保留原分析算法版本，不把 schema 迁移伪装成重新分析。
6. 使用 map v2 字段重算 signature。
7. 删除根级 `bpm / firstBeatMs / barBeatOffset / beatGridMap(v1)`，写入唯一 map v2。
8. 保留 `songStructure`，不在本次迁移中改写或删除。
9. `no-bpm` 等无有效网格状态继续不生成 map。

无法迁移的单曲：

- 保留非网格数据。
- 清除损坏网格。
- 自动来源进入迁移报告中的待分析列表。
- 人工来源进入迁移报告中的待用户处理列表。
- 不阻断其他记录。

### 9.4 Mixtape / Set 去副本

- `mixtape_items.info_json` 删除 BPM、首拍、offset、map、网格 source/version；保留旧段落结果副本。
- `set_items.analysis_json` 删除同类网格副本；保留旧段落结果副本。
- 迁移时不使用这些副本恢复 canonical map。
- 保留项目局部的 `originalBpm`、播放 tempo、gain、转场、loop、key/stem 等非网格字段。

### 9.5 迁移后校验

必须全部通过：

- `PRAGMA integrity_check`。
- 目标表行数与预期一致。
- 所有 v2 map 均可 normalize，fixed/dynamic clip 合法。
- signature 重算一致。
- active JSON 不存在根级旧网格字段。
- Mixtape / Set 不存在网格副本。
- legacy symbol/data 扫描无白名单外命中。
- `user_version = 38`。

全部通过后自动删除临时备份；失败则回滚并保留备份。

## 10. 分析、缓存与人工编辑链路

### 10.1 自动分析

- BeatThis 继续输出并使用 beat/onset 和 downbeat 证据。
- BPM/phase selector 不再保存或比较 exact32 ordinal。
- downbeat lattice 只输出 `0..3`。
- 自动结果直接构建 `source = analysis` 的单 clip map。
- schema 改造本身不要求重跑旧有效网格。
- 如果实现没有改变数值分析行为，不因 map schema v2 单独提升 beat-grid algorithm version；若 selector 数值行为另有变化，必须独立说明和验证。

### 10.2 人工编辑

- 固定与动态编辑都直接修改 map。
- 任一人工提交使 map source 变为 `manual`。
- 后台扫描、补全和普通算法版本检查不得覆盖人工 map。
- 用户主动完整重新分析时先确认；确认后清除旧 map，分析成功才提交新 map。
- 分析失败时保留原人工 map。

### 10.3 分析缓存生命周期

- canonical map 继续属于歌曲分析缓存生命周期。
- 完整重新分析、真正删除歌曲分析缓存或文件永久移除时，map 可以随分析数据清除。
- 仅 waveform/preview/surface 子缓存失效时不得删除 map。
- 歌曲重新加入后重新分析，不自动找回旧人工 map。

## 11. 段落分析边界

本次不改写旧段落算法源码，但阻断其活动生产链路：

- 队列、自动触发和 worker 一律不再请求或执行段落步骤，结果也不得写回；历史 UI 展示暂时保持现状。
- 不删除数据库中的 `songStructure`，不触发段落重算。
- 旧 `songStructure*` 源码和测试原地保留，进入 legacy 白名单，等待后续独立重构。
- 后续独立任务必须先决定旧段落结果是否迁移、作废或重新分析；本草案不预设答案。

## 12. Renderer 与交互规则

### 12.1 网格视觉

- 当前 `beat4` 的颜色、主体宽度和透明度原样继承为 `downbeat`。
- 普通 `beat` 视觉保持不变。
- 删除 32 拍 `bar` 的颜色、宽度、主题变量和绘制分支。
- downbeat 比 beat 朝轨道外侧多突出 3 CSS px。
- 双轨上轨/单轨向上突出，下轨向下突出。
- 外侧空间不足时突出长度限制为可用空间。
- 高 DPI 先以 CSS px 定义，再按 DPR 换算。

### 12.2 密度显隐

- downbeat 在网格开启时保持可见。
- beat 只有在相邻拍线达到共享最小像素间距时绘制。
- 隐藏 beat 只影响绘制，不影响吸附和 transport。
- 阈值由 shared helper 统一，所有 surface 共用。

### 12.3 Clip boundary

- 动态 clip boundary 在所有相关波形中始终显示。
- 使用现有独立 boundary 样式和选中控制帽。
- 不使用 downbeat 的突出规则。
- 与 downbeat 重合时仍须可辨识，建议绘制顺序为：beat -> downbeat -> clip boundary -> selected boundary controls。

### 12.4 吸附与操作

- 原 `bar / beat4` 候选统一为 downbeat。
- 原 beat 精度操作继续逐拍吸附。
- 删除隐藏 32 拍吸附点。
- loop 的 4/8/16/32 beat 只表示长度，不生成网格层级。
- clip boundary 只参与专用动态网格边界编辑。

## 15. 验收标准

### 15.1 数据库

- v35 固定自动网格可无损迁移为 v2 单 clip。
- v35 固定人工网格可无损迁移并保留 source。
- v35 动态多 clip 可逐 clip 迁移。
- 非法单曲不拖死整库，并进入正确报告分组。
- Mixtape/Set 副本被删除且不会回生。
- 高于 36 的数据库被所有入口阻断且零写入。
- 空间不足时零写入。
- 成功迁移后临时备份自动删除；失败时保留并回滚。

### 15.2 网格运行时

- 任何 active runtime line 的 level 只能是 `downbeat` 或 `beat`。
- downbeat 每四拍一次且跨 clip 计算正确。
- fixed/dynamic 使用同一 runtime。
- clip boundary 不进入 line levels。
- active code 不存在 `% 32` 层级生成。

### 15.3 UI 与交互

- 当前四拍线颜色/主体样式保持不变。
- 上轨/单轨向上突出 3px，下轨向下突出 3px。
- light/dark 均正确。
- 全曲压缩时 beat 可隐藏，放大后恢复；downbeat 保持可见。
- clip boundary 始终可见且与 downbeat 可区分。
- downbeat/beat 吸附和 loop 精度符合已确认规则。
- 双轨/编辑模式按既有大波形架构标准完成全量回归。

### 15.4 分析与缓存

- 自动分析生成 analysis 单 clip map。
- 后台任务不覆盖 manual map。
- 完整重新分析有确认，成功替换、失败保留。
- waveform 子缓存清理不删除 map。
- 完整分析缓存清理会删除 map，并可重新分析恢复初始结果。
- 旧段落分析的任务入队、自动触发、worker 执行和结果持久化必须阻断；历史结果是否继续展示不在本次重构中改写，待段落功能单独立项时统一决定。

### 15.5 Benchmark

- strict pass 继续检查 downbeat mod4。
- exact32 不进入新报告或 selector。
- candidate oracle、selected fail 和分类统计仍可生成。
- 历史原件 hash 不变化。

## 17. 已接受风险与后续债务

- 当前版本以前已经发布的软件没有高版本数据库守卫；用户确认当前无需处理历史用户。
